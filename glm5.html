1. 核心范式 — 从 Vibe Coding 到 Agentic Engineering + ARC 三合一

Vibe Coding = 向模型要一段代码片段然后粘贴;Agentic Engineering = 把整个 repo + 一个 terminal 交给模型,放手一小时让它自主完成端到端软件工程任务(读代码库 → 定位 issue → 改多文件 → 跑测试 → 迭代)。GLM-5 就是为后者而生:延续 GLM-4.5 把 Agentic / Reasoning / Coding (ARC) 三类能力统一进单个 MoE 的路线,并把重心从"被动知识库"推向"主动问题求解器"。

规模上 GLM-5 扩到 256 experts,同时把层数降到 80 层以减少 expert parallelism 的通信开销,得到 744B 总参 / 40B 激活(相比 GLM-4.5 的 355B/32B 翻倍总量)。预训练 28.5T tokens,上下文分三段渐进扩展:32K (1T) → 128K (500B) → 200K (50B)。SFT 起支持三种思考特性:Interleaved Thinking(每次回复/工具调用前都思考)、Preserved Thinking(coding agent 多轮里保留全部 thinking 块、复用推理不重推)、Turn-level Thinking(逐轮开关思考,轻任务省延迟、重任务开思考)。

# 两种范式对比

# ❌ Vibe Coding (被动)

model: "def quicksort(a): ..."
# 人负责: 集成 / 调试 / 跑测试

# ✅ Agentic Engineering (自主)

task = "修复 issue #1234, 让测试通过"
while not done:
    thought = model.think(ctx)     # 规划
    action  = model.act(ctx)       # bash/edit/read
    obs     = env.run(action)      # F2P/P2P 测试反馈
    ctx = update(ctx, obs)
# 人只在最后 review PR

# ARC 三合一进单个 MoE:
#   Agentic + Reasoning + Coding
# 744B total / 40B active · 256 experts · 80 层

核心主张

Vibe → Agentic: LLM 的价值从"给你一段代码"升级为"独立完成一整个工程任务"。评测也随之从片段正确率转向端到端的 Build/Instance/Check 成功率。
256 experts + 80 层: 反常识地减层增专家——层数少 → EP 跨设备通信轮次少;专家多 → 总容量大而激活稀疏。744B 总参但每 token 只激活 40B。
200K 上下文: 比 GLM-4.5 的 128K 再进一步,专为超长文档 + 多文件代码库设计,是 agentic engineering "读整个 repo" 的前提。
Preserved Thinking: 借鉴 Claude Opus 4.5,coding agent 多轮里不丢弃历史 thinking,避免每轮重新推导 → 长程任务上信息一致、更稳。

2. 架构四件套 — MLA+Muon Split · MTP 参数共享 · DSA 稀疏注意力

① MLA + Muon Split: Multi-latent Attention (MLA) 用低秩 KV 省显存、加速长文,但实验发现 576 维 latent KV 的 MLA 在 Muon 优化器下打不过 GQA-8(2048 维 KV)。解法是改造 Muon:不再对整块上投影矩阵 $W^{UQ},W^{UK},W^{UV}$ 做正交化,而是按 head 拆成小矩阵分别正交化(Muon Split),让不同 head 的投影权重以不同尺度更新 → 补齐到 GQA-8 水平,且训练中 attention logits 天然稳定、无需 clipping。② MLA-256: 把 head dim 从 192 提到 256、head 数减 1/3,保持训练算力/参数量不变的同时降低解码算力。③ MTP 参数共享: 训练时共享 3 个 MTP 层参数(而非 DeepSeek-V3 单层),draft 模型显存与 DSV3 持平但提升接受率——accept length 2.76 > DSV3.2 的 2.55。

④ DSA (DeepSeek Sparse Attention): 把稠密 $O(L^2)$ 注意力换成内容感知的细粒度 token 选择(不是滑窗那种固定模式,而是"看内容"决定哪些 token 重要)。关键是不从头训——从稠密 base 做 Continued Pre-Training:先 dense warm-up(1000 步,每步 14 条 202,752-token 序列,LR 5e-3),再 sparse adaptation(仅 20B tokens,远小于 DSV3.2 的 943.7B)。长文 benchmark(MQ/MV-NIAH-128k、HotpotQA)≈ MLA,而长序列注意力算力降 1.5–2× → 128K 上下文用一半 GPU 成本跑。

# ① Muon Split: 按 head 独立正交化
# 原始: 整块 W^UQ/W^UK/W^UV 一起正交化
# Split: 拆成各 head 的小矩阵分别做
for h in heads:
    W_h = orthogonalize(W_up[h])   # 各头独立尺度
# → MLA 追平 GQA-8, logits 稳定无需 clip

# ② MLA-256: head_dim 192→256, heads×2/3
#   训练算力/参数不变, 解码算力↓

# ③ MTP 参数共享 (预测下 3 token)
mtp_layers = share_params(n=3)   # 显存持平 DSV3
# accept_len: GLM-5 2.76 > DSV3.2 2.55

# ④ DSA Continued Pre-Training (两阶段)
warmup   = train(steps=1000, seq=202752, lr=5e-3)
sparse_ad = train(tokens=20e9)      # ≪ DSV3.2 943.7B
# dynamic top-k token 选择, 非固定滑窗
# 长文≈MLA, attn 算力 ↓1.5-2× @128K

为什么这么设计

Muon Split 补 MLA 的坑: MLA 省显存但精度掉;整块正交化让各 head 被迫同尺度更新。按 head 拆开后各自适配 → 追平 GQA-8,还顺带稳住了 attention logits。
DSA 走 continued pretraining: 从头训稀疏注意力成本天文数字。从稠密 base 只花 20B tokens 适配(DSV3.2 用了 943.7B),就证明"长文里 90% 注意力项是冗余的"。
MLA-256 是硬件考量: DSV3 的 head 数按 H800 roofline 选,换硬件就不合适。加大 head dim、减 head 数,在不动训练成本的前提下砍解码计算。
MTP 参数共享: 训练用 3 层预测下 3 token 但共享参数 → 显存不涨、消除训推 draft 不一致,推测解码接受率反超 DSV3.2。

3. 预训练 & Mid-training — 数据构造与 28.5T 预算分配

GLM-5 base 分两大阶段:Pre-training(通用语言 + 代码,建知识底座)与 Mid-training(拉长上下文 + 灌 agentic/长文能力),总预算 28.5T tokens(pre-training 约 27T,mid-training 约 1.55T)。预训练数据三大源: ① Web — 在 GLM-4.5 管线上新增基于句向量的 DCLM 分类器捞高质量数据,再用 World Knowledge 分类器(Wikipedia + LLM 标注训练)从中低质数据里蒸馏长尾知识;② Code — 刷新代码托管平台快照 + 更多含码网页,fuzzy-dedup 后唯一 token 增加 28%,修 Software Heritage 元数据对齐、给低资源语言(Scala/Swift/Lua)单独训分类器;③ Math/Science — 用 LLM 打教育性分,长文档用 chunk-and-aggregate 提升打分精度,严格排除合成/AI 生成/模板数据。

Mid-training 结构: 上下文三段渐进 32K (1T) → 128K (500B) → 200K (50B),长文档与合成 agent 轨迹在后段上采样。SWE 数据: 沿用"repo 级代码文件 + commit diff + issue + PR + 相关源文件拼成统一序列"的范式,放宽 repo 级过滤扩大候选池 → 约 10M issue–PR 对,同时强化单 issue 质量过滤降噪,并为每对检索更多相关文件;过滤后 issue–PR 部分约 160B 唯一 token。长文数据: 自然数据(书/论文/文档,PPL+去重+长度多级过滤,知识密集域上采样)+ 合成数据(借鉴 NextLong/EntropyLong,把高相似文本 interleaved packing 构造长程依赖抗 lost-in-the-middle;200K 段加少量 MRCR-like 多变体强化多轮召回)。训练用 Muon,LR:pre-train 0→2e-4 warmup 后 cos decay 到 4e-5,mid-training 线性 4e-5→1e-5。

数据构造要点

World Knowledge 分类器: 不是简单丢掉中低质数据,而是用 Wikipedia+LLM 标注训分类器,从"垃圾堆"里精捞长尾知识——扩大有效知识覆盖而非只提纯。
Math 严禁合成: 数学/科学数据坚决不用合成/AI 生成/模板数据,只保留 LLM 判为高教育性的真实内容。抵制"合成数据污染推理"的风险。
SWE 放宽 repo/收紧 issue: repo 级过滤放松以扩池到 ~10M 对,但单 issue 质量收紧降噪——广度靠 repo,信噪比靠 issue 双向调。
200K 反哺 128K: 在 128K 基础上再加 200K 段,连 128K 窗口内的表现都变强——长上下文训练的收益能"下溢"到短窗口。

4. SFT — 多任务对齐 · 三种思考特性 · 错误段 masking

Post-training 第一步是多任务 SFT,把 base 变成会思考、会用工具的助手。相比 GLM-4.5,GLM-5 大幅扩充 Agent 与 Coding 数据规模。SFT 语料三大类: ① General Chat(QA、写作、角色扮演、翻译、多轮、长文交互);② Reasoning(数学/编程/科学推理);③ Coding Agent(前后端工程代码、tool calling、coding agent、search agent、通用 agent)。上下文在 SFT 阶段拉到 202,752 tokens,配新 chat template 支持三种思考特性。

数据构造(按类别): General Chat 优化为更精炼的回复风格,角色扮演按 instruction following / 表达力 / 创造力 / 逻辑一致性 / 长对话一致性 五维自动+人工过滤。Reasoning 中逻辑题构造可验证问题 + 拒绝采样合成高质数据;数学/科学做难度过滤,只留 GLM-4.7 都觉难的题。Coding/Agent 大量构建执行环境获取高质轨迹(重真实场景 + 长程任务),再用 expert RL + 拒绝采样精修。关键技巧:轨迹里的错误段落保留但在 loss 中 mask 掉 → 让模型学会"纠错行为"而不强化错误动作。此外 SFT 阶段应用 INT4 QAT(量化感知训练),配自研量化 kernel 保证训推 bitwise 一致。

SFT 的巧思

错误段 masking: 不删含错轨迹,而是把错误 token 的 loss 置零、保留在上下文中。模型看到"错→纠"的完整序列却只学纠正动作 → 天然习得自我纠错,这是 agent 长程任务的关键能力。
难度过滤: 数学/科学只留 GLM-4.7 都难的题,避免用简单题浪费 SFT 预算——把监督信号集中在"能力边界"上。
三种思考特性: Interleaved 提指令遵循;Preserved 借鉴 Opus 4.5,多轮不丢 thinking;Turn-level 让轻任务省延迟、重任务开思考——延迟与精度可控权衡。
SFT 阶段就上 QAT: INT4 量化感知训练放在 SFT 而非事后,配 bitwise 一致 kernel,低精度部署几乎无损。

5. Reasoning RL — GRPO + IcePop (去 KL) · 四域混训 · DSA RL 洞见

算法骨架: 在 GRPO 上叠加 IcePop 缓解训推失配(采样用的 inference policy 与更新用的 training policy 分布不一致)。显式区分 $\pi^{\text{train}}$(算梯度)与 $\pi^{\text{infer}}$(采轨迹),并移除 KL 正则项以加速 RL 提升。核心是 $\operatorname{pop}$ 算子:当训推失配比 $\rho_{i,t}$ 偏离太远就把该 token 的贡献直接置零,只在 $[1/\beta,\beta]$ 信任区内正常回传。超参 $\beta=2,\ \epsilon_{\text{low}}=0.2,\ \epsilon_{\text{high}}=0.28$,完全 on-policy,group size 32、batch size 32。

$$\mathcal{L}(\theta)=-\mathbb{E}_{x,\{y_i\}\sim\pi^{\text{infer}}_{\theta_{old}}}\Big[\tfrac{1}{G}\textstyle\sum_{i=1}^{G}\tfrac{1}{|y_i|}\sum_{t=1}^{|y_i|}\operatorname{pop}(\rho_{i,t},1/\beta,\beta)\cdot\min\big(r_{i,t}\hat{A}_{i,t},\ \operatorname{clip}(r_{i,t},1-\epsilon_{low},1+\epsilon_{high})\hat{A}_{i,t}\big)\Big]$$

其中失配比 $\rho_{i,t}=\pi^{\text{train}}_{\theta_{old}}/\pi^{\text{infer}}_{\theta_{old}}$,PPO 比率 $r_{i,t}=\pi^{\text{train}}_\theta/\pi^{\text{train}}_{\theta_{old}}$,组归一化优势 $\hat{A}_{i,t}=(R_i-\operatorname{mean}(R))/\operatorname{std}(R)$,而 $\operatorname{pop}(\rho,1/\beta,\beta)=\rho$ 当 $1/\beta\le\rho\le\beta$,否则 $0$。四域混训: 数学 / 科学 / 代码 / 工具集成推理 (TIR)。数学科学做难度过滤——只留 GLM-4.7 极少答对或答不出、但更强 teacher(GPT-5.2 xhigh / Gemini 3 Pro)能解的题;代码来自 Codeforces / TACO / SYNTHETIC-2-RL(竞赛)+ 内部拆解到最小实现(科学编码);TIR 复用数学科学难子集 + 与标注商共建"必须用工具"的 STEM 题。各域专属 judge 出二元 outcome reward,四域大致均衡混合。DSA RL 洞见: indexer 的 top-k($k{=}2048$)必须确定性——用 torch.topk(略慢但确定)而非 SGLang 的非确定 CUDA top-k,否则几步后熵崩、性能骤降;RL 中默认冻结 indexer 参数。

# GRPO + IcePop (去 KL), 完全 on-policy
# 区分两个 policy:
pi_train   # 算梯度
pi_infer   # 采样轨迹 → 二者分布有失配

rho = pi_train_old / pi_infer_old   # 训推失配比
r   = pi_train / pi_train_old       # PPO 比率
A   = (R - mean(R)) / std(R)        # 组归一化优势

def pop(rho, beta):        # β=2
    return rho if (1/beta <= rho <= beta) else 0
# 失配太大 → 该 token 贡献置零 (比裁剪更硬)

loss = -mean( pop(rho,beta) *
        min(r*A, clip(r, 1-0.2, 1+0.28)*A) )
# group_size=32, batch_size=32

# 四域混训 + 二元 reward
domains = [math, science, code, TIR]
math = filter(hard_for_glm47 & solvable_by_gpt52)
reward = domain_judge(traj)   # binary outcome

# DSA RL: indexer top-k 必须确定性
idx = torch.topk(scores, k=2048)  # 非 CUDA topk
freeze(indexer)               # RL 中冻结

Reasoning RL 关键

pop 比 clip 更硬: PPO 的 clip 只是限制梯度幅度,pop 直接把失配超界的 token 置零。异步/训推分布不一致时,这能挡住噪声梯度污染更新。
去掉 KL: IcePop 原版有 KL 正则,GLM-5 移除以加速 RL 提升——pop 门控已提供稳定性,KL 反而拖慢收敛。
难度过滤定位能力边界: 只训"自己难、更强模型可解"的题——太简单没增益,太难无正样本。这是把 RL 预算花在刀刃上的通用配方。
确定性 top-k 是 DSA RL 命门: indexer 选 token 若非确定,训推选中的 token 不同 → 类似 MoE 路由不一致,熵崩。用 torch.topk + 冻结 indexer 换来稳定。

6. Agentic RL 基础设施 — 全异步解耦 · slime · TITO · 双侧重要性采样

长程 agentic 任务的 rollout 是重长尾的(有的轨迹几步就结束,有的要几百步),同步 RL 会因等最慢的轨迹产生大量 GPU bubble。GLM-5 用全异步解耦:训练引擎与推理引擎分处不同 GPU,推理引擎持续生成轨迹,攒够阈值就打包送训练引擎更新;每 $K$ 次梯度更新把新权重推回推理引擎,并重置优化器以应对 rollout policy 变化。算法用 group-wise policy optimization,对每个问题 $x$ 采 $K$ 条轨迹,以组内均值为 baseline:$L(\theta)=\mathbb{E}_{x}\big[\frac1K\sum_i(r(x,y_i)-\bar r(x))\big]$,且只对模型生成的 token 算 loss,环境反馈不计入。

异步带来严重 off-policy 问题(一条轨迹可能由多个模型版本生成)。两个关键工程:① TITO Gateway (Token-in-Token-out): 直接消费推理引擎产出的原始 token 流,而非把文本 re-tokenize——避免 token 边界/空白/特殊符错位破坏 action-reward 对齐。② Direct 双侧重要性采样: 精确追踪 $\pi_{\theta_{old}}$ 需要存海量历史 checkpoint,不现实;于是用 rollout 时的 log-prob 作行为代理 $r_t(\theta)=\pi_\theta/\pi_{rollout}$(丢弃 $\pi_{\theta_{old}}$),并做双侧对称 clip:落在 $[1-\epsilon_\ell,\,1+\epsilon_h]$ 之外的 token 直接 mask 出梯度(比 PPO 非对称裁剪更稳,类似 IcePop 但更简)。多任务侧由 slime 的 Server-based Multi-Task Rollout Orchestrator 调度,各任务以微服务注册、统一 message-list 表示,支持 >1k 并发 rollout + 动态采样比。

# 全异步解耦: 推理/训练分 GPU
while True:                         # 推理引擎 (持续)
    traj = rollout(pi_rollout, env)   # 长尾轨迹
    buffer.push(traj)                 # TITO: 存 token IDs

if len(buffer) >= threshold:          # 训练引擎
    batch = buffer.drain()
    theta = update(theta, batch)      # 每 K 步...
    push_weights(theta -> rollout)    # ...同步权重
    reset_optimizer()                 # 应对 policy 漂移

# group-wise 优势 (只算模型 token)
# L = E_x[ 1/K Σ_i (r_i - mean(r)) ]

# 双侧对称 clip 的 IS ratio
r_t = exp(logp_theta - logp_rollout)  # 丢 π_old
if r_t < 1-eps_l or r_t > 1+eps_h:
    mask(token)                       # 出界 → 不回传梯度

异步 RL 的三个坑

长尾 → bubble: agentic 轨迹长度极不均衡,同步训练卡在最慢那条。解耦推理/训练到不同 GPU,推理不停生成、训练攒批更新 → GPU 利用率拉满。
TITO 防错位: 若把 rollout 文本 re-tokenize,token 边界/空白/特殊符会漂,破坏 action↔reward 的步级对齐。直接吃原始 token 流,保证采样即所学。
丢弃 π_old: 异步下一条轨迹跨多个模型版本,精确 π_old 要存一堆 checkpoint。改用 rollout log-prob 当代理,消掉一次额外推理。
双侧对称 clip: 不同于 PPO 单侧裁剪,落在信任区外的 token 直接 mask 掉梯度,防极端 policy 偏离引发的不稳定——比 IcePop 更简。

7. General RL + On-Policy 跨阶段蒸馏 — 人味对齐与防遗忘

General RL 结构: 把优化目标拆成三个互补维度 —— ① Foundational Correctness(基础正确性:指令遵循失败、逻辑矛盾、事实错误、知识幻觉、语言不流畅,把错误率压到可用基线,是一切后续优化的前提);② Emotional Intelligence(情商:共情、有洞察、接近自然人类交流);③ Task-Specific Quality(任务专项质量:写作、文本处理、主客观 QA、角色扮演、翻译,从"正确"提到"高质")。数据/奖励构造: 用混合奖励系统组合三类信号 —— rule-based(精确可解释但只覆盖可规则化的)、ORM 输出奖励模型(低方差高效但易 reward hacking)、GRM 生成式奖励模型(抗刷分但方差高),三者互补取长补短。人在环风格对齐: 显式引入高质量人写回复作为风格/质量锚点,纠正纯模型优化收敛到的"模型味"(啰嗦、公式化、缺人类文笔的细腻)。

On-Policy 跨阶段蒸馏(最后一步,防遗忘): 多阶段顺序 RL 会累积遗忘早先能力,故最终阶段做 on-policy 蒸馏,快速找回 SFT / Reasoning RL / General RL 阶段的技能。结构: 前面各阶段的最终 checkpoint 当 teacher,prompt 从对应 teacher 的 RL 训练集按比例混采;把 Eq.1 里的优势项替换为 teacher 与 student 的 log 概率比(sg = stop gradient / detach):

$$\hat{A}_{i,t}=\operatorname{sg}\!\Big[\log\frac{\pi^{\text{infer}}_{\theta_{\text{teacher}}}(y_{i,t}\mid x,y_{i,\lt t})}{\pi^{\text{train}}_{\theta}(y_{i,t}\mid x,y_{i,\lt t})}\Big]$$

为什么要这两步

三维分层: 基础正确性是地基(错了再漂亮也误导),情商与专项质量是上层建筑。先保证可用,再谈好用——目标解耦让奖励设计各司其职。
三奖励互补: rule-based 精确但窄、ORM 高效但易被 hack、GRM 抗 hack 但方差大。单用任何一个都翻车,混合才能兼顾精度/效率/鲁棒。
人写回复当锚: 纯模型自优化会收敛到"模型腔"(啰嗦、模板化)。注入专家人类回复作风格锚点,把分布往"自然人类表达"拉。
跨阶段蒸馏防遗忘: 顺序做 Reasoning→Agentic→General RL 会冲淡早先能力。最后用各阶段 final ckpt 当 teacher 做 on-policy 蒸馏,用 log 概率比当优势,快速找回全部技能。
