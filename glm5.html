1. 核心范式 — 从 Vibe Coding 到 Agentic Engineering + ARC 三合一

Vibe Coding = 向模型要一段代码片段然后粘贴;Agentic Engineering = 把整个 repo + 一个 terminal 交给模型,放手一小时让它自主完成端到端软件工程任务(读代码库 → 定位 issue → 改多文件 → 跑测试 → 迭代)。GLM-5 就是为后者而生:延续 GLM-4.5 把 Agentic / Reasoning / Coding (ARC) 三类能力统一进单个 MoE 的路线,并把重心从"被动知识库"推向"主动问题求解器"。

规模上 GLM-5 扩到 256 experts,同时把层数降到 80 层以减少 expert parallelism 的通信开销,得到 744B 总参 / 40B 激活(相比 GLM-4.5 的 355B/32B 翻倍总量)。预训练 28.5T tokens,上下文分三段渐进扩展:32K (1T) → 128K (500B) → 200K (50B)。SFT 起支持三种思考特性:Interleaved Thinking(每次回复/工具调用前都思考)、Preserved Thinking(coding agent 多轮里保留全部 thinking 块、复用推理不重推)、Turn-level Thinking(逐轮开关思考,轻任务省延迟、重任务开思考)。

核心主张

Vibe → Agentic: LLM 的价值从"给你一段代码"升级为"独立完成一整个工程任务"。评测也随之从片段正确率转向端到端的 Build/Instance/Check 成功率。
256 experts + 80 层: 反常识地减层增专家——层数少 → EP 跨设备通信轮次少;专家多 → 总容量大而激活稀疏。744B 总参但每 token 只激活 40B。
200K 上下文: 比 GLM-4.5 的 128K 再进一步,专为超长文档 + 多文件代码库设计,是 agentic engineering "读整个 repo" 的前提。
Preserved Thinking: 借鉴 Claude Opus 4.5,coding agent 多轮里不丢弃历史 thinking,避免每轮重新推导 → 长程任务上信息一致、更稳。

2. 架构四件套 — MLA+Muon Split · MTP 参数共享 · DSA 稀疏注意力

① MLA + Muon Split: Multi-latent Attention (MLA) 用低秩 KV 省显存、加速长文,但实验发现 576 维 latent KV 的 MLA 在 Muon 优化器下打不过 GQA-8(2048 维 KV)。解法是改造 Muon:不再对整块上投影矩阵 $W^{UQ},W^{UK},W^{UV}$ 做正交化,而是按 head 拆成小矩阵分别正交化(Muon Split),让不同 head 的投影权重以不同尺度更新 → 补齐到 GQA-8 水平,且训练中 attention logits 天然稳定、无需 clipping。② MLA-256: 把 head dim 从 192 提到 256、head 数减 1/3,保持训练算力/参数量不变的同时降低解码算力。③ MTP 参数共享: 训练时共享 3 个 MTP 层参数(而非 DeepSeek-V3 单层),draft 模型显存与 DSV3 持平但提升接受率——accept length 2.76 > DSV3.2 的 2.55。

④ DSA (DeepSeek Sparse Attention): 把稠密 $O(L^2)$ 注意力换成内容感知的细粒度 token 选择(不是滑窗那种固定模式,而是"看内容"决定哪些 token 重要)。关键是不从头训——从稠密 base 做 Continued Pre-Training:先 dense warm-up(1000 步,每步 14 条 202,752-token 序列,LR 5e-3),再 sparse adaptation(仅 20B tokens,远小于 DSV3.2 的 943.7B)。长文 benchmark(MQ/MV-NIAH-128k、HotpotQA)≈ MLA,而长序列注意力算力降 1.5–2× → 128K 上下文用一半 GPU 成本跑。

为什么这么设计

Muon Split 补 MLA 的坑: MLA 省显存但精度掉;整块正交化让各 head 被迫同尺度更新。按 head 拆开后各自适配 → 追平 GQA-8,还顺带稳住了 attention logits。
DSA 走 continued pretraining: 从头训稀疏注意力成本天文数字。从稠密 base 只花 20B tokens 适配(DSV3.2 用了 943.7B),就证明"长文里 90% 注意力项是冗余的"。
MLA-256 是硬件考量: DSV3 的 head 数按 H800 roofline 选,换硬件就不合适。加大 head dim、减 head 数,在不动训练成本的前提下砍解码计算。
MTP 参数共享: 训练用 3 层预测下 3 token 但共享参数 → 显存不涨、消除训推 draft 不一致,推测解码接受率反超 DSV3.2。

3. Agentic RL 基础设施 — 全异步解耦 · slime · TITO · 双侧重要性采样

长程 agentic 任务的 rollout 是重长尾的(有的轨迹几步就结束,有的要几百步),同步 RL 会因等最慢的轨迹产生大量 GPU bubble。GLM-5 用全异步解耦:训练引擎与推理引擎分处不同 GPU,推理引擎持续生成轨迹,攒够阈值就打包送训练引擎更新;每 $K$ 次梯度更新把新权重推回推理引擎,并重置优化器以应对 rollout policy 变化。算法用 group-wise policy optimization,对每个问题 $x$ 采 $K$ 条轨迹,以组内均值为 baseline:$L(\theta)=\mathbb{E}_{x}\big[\frac1K\sum_i(r(x,y_i)-\bar r(x))\big]$,且只对模型生成的 token 算 loss,环境反馈不计入。

异步带来严重 off-policy 问题(一条轨迹可能由多个模型版本生成)。两个关键工程:① TITO Gateway (Token-in-Token-out): 直接消费推理引擎产出的原始 token 流,而非把文本 re-tokenize——避免 token 边界/空白/特殊符错位破坏 action-reward 对齐。② Direct 双侧重要性采样: 精确追踪 $\pi_{\theta_{old}}$ 需要存海量历史 checkpoint,不现实;于是用 rollout 时的 log-prob 作行为代理 $r_t(\theta)=\pi_\theta/\pi_{rollout}$(丢弃 $\pi_{\theta_{old}}$),并做双侧对称 clip:落在 $[1-\epsilon_\ell,\,1+\epsilon_h]$ 之外的 token 直接 mask 出梯度(比 PPO 非对称裁剪更稳,类似 IcePop 但更简)。多任务侧由 slime 的 Server-based Multi-Task Rollout Orchestrator 调度,各任务以微服务注册、统一 message-list 表示,支持 >1k 并发 rollout + 动态采样比。

# 全异步解耦: 推理/训练分 GPU
while True:                         # 推理引擎 (持续)
    traj = rollout(pi_rollout, env)   # 长尾轨迹
    buffer.push(traj)                 # TITO: 存 token IDs

if len(buffer) >= threshold:          # 训练引擎
    batch = buffer.drain()
    theta = update(theta, batch)      # 每 K 步...
    push_weights(theta -> rollout)    # ...同步权重
    reset_optimizer()                 # 应对 policy 漂移

异步 RL 的三个坑

长尾 → bubble: agentic 轨迹长度极不均衡,同步训练卡在最慢那条。解耦推理/训练到不同 GPU,推理不停生成、训练攒批更新 → GPU 利用率拉满。
TITO 防错位: 若把 rollout 文本 re-tokenize,token 边界/空白/特殊符会漂,破坏 action↔reward 的步级对齐。直接吃原始 token 流,保证采样即所学。
丢弃 π_old: 异步下一条轨迹跨多个模型版本,精确 π_old 要存一堆 checkpoint。改用 rollout log-prob 当代理,消掉一次额外推理。
双侧对称 clip: 不同于 PPO 单侧裁剪,落在信任区外的 token 直接 mask 掉梯度,防极端 policy 偏离引发的不稳定——比 IcePop 更简。
