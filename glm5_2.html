核心发现

通过对比 HuggingFace 上 zai-org/GLM-5.1 和 zai-org/GLM-5.2 的 config.json，仅有三处差异：

① Context 5x: max_position_embeddings 从 202,752 → 1,048,576 (1M tokens)。这是 GLM-5.2 的核心卖点。
② RoPE θ 8x: rope_theta 从 1M → 8M。经典的 ABF (Adjusted Base Frequency) 长上下文扩展手法——放大 θ 让高频分量覆盖更长距离。
③ DSA Indexer 优化: 新增 index_topk_freq=4 (每4层做一次 full index, 其余层共享)、index_share_for_mtp_iteration=true (MTP 迭代间共享 indexer 节省计算)、index_skip_topk_offset=3 (前3层跳过 sparse index 用 dense attn)。

架构层面 (MLA-256, 256+1 MoE Top-8, 78 layers, hidden 6144, vocab 154880) 完全相同。

2. DSA Indexer 共享优化 — 1M Context 的计算效率关键

// DSA (DeepSeek Sparse Attention) 回顾
// 核心: 不是每个 Q 都看所有 KV
// 而是用 Indexer 选出 top-k 个
// 最相关的 KV chunk 来做 attention

// GLM-5.1 DSA Indexer 配置:
//   index_head_dim: 128
//   index_n_heads: 32
//   index_topk: 2048
//   indexer_types: ["full","full","full",
//     "shared","shared","shared", ...]
//   (每4层: 1个full + 3个shared)

// GLM-5.2 新增优化:
index_topk_freq: 4
// 每4层才重新计算 full index
// 中间层复用上一次 index 结果
// 计算量: indexer FLOPS 降至 1/4

index_share_for_mtp_iteration: true
// MTP (Multi-Token Prediction) 的
// 多个预测迭代共享同一份 index
// 避免重复计算 sparse pattern

index_skip_topk_offset: 3
// 前3层 (first_k_dense_replace=3)
// 本来就是 dense attention
// 跳过 topk 计算 (无意义)

// ═══════════════════════════════════
// indexer_types 模式 (78层):
// [full, full, full,  ← 前3层dense
//  shared, shared, shared,
//  full, shared, shared, shared,
//  full, shared, shared, shared,
//  ...重复到78层]
// 规律: 每4层1个full indexer
// full层计算新index, shared层复用

为什么需要 Indexer 共享？

DSA 的核心是用 Indexer 为每个 Q 选出最相关的 top-k KV chunks（GLM 用 top-2048）。在 200K context 下 indexer 成本可控，但扩展到 1M context 后：

• 候选 KV chunks 数量 5x → indexer 计算量 5x

• 78 层 × 每层独立计算 → 巨大冗余

• MTP 3 次迭代 × 每次重算 → 3x 浪费

三层优化策略

层间共享 (topk_freq=4): 相邻层的 attention pattern 高度相似（实验证明相关性 >90%）。每4层只让1层做 full indexer 计算，其余3层直接复用。Indexer FLOPS 降至 1/4。
MTP 迭代共享: Multi-Token Prediction 的多个预测 head 看到的 context 几乎相同（只多了 1-2 个 token）。共享 index 完全合理。
Dense 层跳过: 前3层本来就是 full attention（first_k_dense_replace=3），做 sparse index 计算是浪费，直接跳过。

这三个优化使 GLM-5.2 在 1M context 下的 DSA indexer 开销仅为原始的 ~25%，是支撑 "真正可用的 1M" 的关键工程。

// 核心公式:
//   θ_new = θ_old × scale_factor

// 原理: 放大 θ 使旋转角变慢

长上下文扩展技术

RoPE θ 放大是最经典的长上下文扩展方法。GLM-5.2 将 θ 从 1M 放大到 8M（8x），配合 max_position_embeddings 从 200K 扩展到 1M（5x）。θ 的放大倍数大于 context 扩展倍数，给出余量。

这与 DeepSeek-V3 (θ=10M for 128K) 和 Qwen2.5 (θ=1M for 131K) 的策略一致。
