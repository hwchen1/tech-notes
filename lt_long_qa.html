1. LongCite — 精确引用定位

# ====== LongCite: 精确引用定位 ======
# Core: 模型回答长文本问题时
# 必须同时输出 fine-grained citation
# 精确到句子/段落级别

# ====== Citation 格式 ======
# 回答中每个 claim 必须附带来源标注:
"The transformer architecture was introduced
 in 2017 [cite: doc_3, 142, 289].
 It uses self-attention mechanisms
 [cite: doc_3, 290, 445] to process
 sequences in parallel."

# 格式: [cite: doc_id, start_pos, end_pos]
# doc_id: 文档编号
# start_pos: 引用起始字符位置
# end_pos: 引用结束字符位置

# ====== 两阶段训练 Pipeline ======
#
# Stage 1: CoarseCite (粗粒度)
#   - 文档级引用: [cite: doc_id]
#   - 训练数据量大, 自动构建
#   - 让模型学会"哪篇文档相关"
#
# Stage 2: FineCite (细粒度)
#   - 精确位置: [cite: doc_id, start, end]
#   - 人工验证 + LLM辅助标注
#   - 让模型学会"哪个句子支撑 claim"
#
# ====== 评测三维度 ======
# Citation Recall: 有多少 claim 被引用覆盖
# Citation Precision: 引用是否真的支撑 claim
# Answer Quality: 答案本身的正确性和完整性

# ====== 开源资源 ======
# LongCite-45K 数据集
# GLM-4-9B-LongCite 模型
# 支持 128K 上下文长度

LongCite 两阶段训练: 粗粒度 doc 级引用 -> 细粒度句子级引用, 三维评测体系

核心意义

长文本 QA 最大的问题: 用户无法验证答案是否来自上下文。LongCite 让每个 claim 都可追溯, 从"信任模型"变为"验证模型"。

核心创新：将 citation generation 从后处理提升为模型的原生能力, 通过两阶段训练 (CoarseCite -> FineCite) 逐步提升定位精度。

为什么需要两阶段?

直接训练 FineCite 难度太大 -- 模型需要同时学"找到相关文档"和"定位精确位置"。先用大量粗粒度数据建立文档级感知, 再用精标数据训练位置定位。

关键参数

基座模型GLM-4-9B
训练数据LongCite-45K
上下文长度128K tokens
引用粒度句子/段落级
评测维度Recall + Precision + Quality

应用场景

法律文档分析、学术论文综述、企业知识库问答 -- 所有需要"有据可查"的场景。

2. LongFaith — 忠实度保证

# ====== LongFaith: 忠实度保证 ======
# Core: 长上下文回答的 faithfulness
# 不 hallucinate, 不编造超出上下文的信息

# ====== 核心问题 ======
# 上下文越长, 模型越容易 hallucinate
# 原因: 注意力稀释 (attention dilution)
# 128K context → 注意力分散 → 信息模糊
# → 模型"补脑"编造看似合理的内容

# ====== 三大策略 ======

# Strategy 1: Faithful Decoding
class FaithfulDecoder:
    """约束解码: 只生成上下文中有支撑的内容"""
    def decode(self, context, query):
        # 每生成一个 token, 检查是否有证据
        # 若无支撑 → 降低该 token 概率
        # 强制模型"有证据才说话"
        logits = self.model(context, query)
        support = self.check_support(logits, context)
        return logits * support  # mask unsupported

# Strategy 2: Citation-Guided Generation
class CitationGuided:
    """先定位证据, 再生成回答"""
    def generate(self, context, query):
        # Step 1: 检索相关片段
        evidence = self.locate_evidence(context, query)
        # Step 2: 仅基于证据生成
        answer = self.model(evidence, query)
        return answer  # grounded in evidence

# Strategy 3: Self-Verification
class SelfVerifier:
    """生成后自我验证每个 claim"""
    def verify(self, answer, context):
        claims = self.extract_claims(answer)
        for claim in claims:
            supported = self.nli(claim, context)
            if not supported:
                claim.flag = "UNSUPPORTED"
        return self.revise(answer, claims)

# ====== 与 RAG 的区别 ======
# RAG: 外部检索 → 注入上下文 → 生成
# LongFaith: 上下文已给定 → 保持忠实
# 本质: RAG 管"找信息", LongFaith 管"不编造"

LongFaith 核心循环: 生成 -> claim 抽取 -> 对比上下文验证 -> 标记/移除无支撑内容

核心问题

上下文越长, hallucination 越严重。128K 上下文中, 模型注意力被稀释, 对远距离信息的记忆变得模糊, 容易"脑补"不存在的内容。

Attention Dilution：当 context length 从 4K 增长到 128K, 每个 position 能分到的注意力权重下降 \(\sim 32\times\), 导致远处信息的 recall 显著降低。

三策略对比

策略时机代价
Faithful Decoding生成时速度慢 (逐 token 检查)
Citation-Guided生成前需要额外检索步骤
Self-Verification生成后需要二次推理

与 RAG 的本质区别

RAG: 从外部检索信息注入上下文 -- 解决"没有信息"的问题。
LongFaith: 在已给定的长上下文内保持忠实 -- 解决"有信息但乱说"的问题。

评测维度：Faithfulness Score (忠实度分数), Hallucination Rate (幻觉率), Coverage (信息覆盖率) -- 三者需要平衡, 过度保守会损失 coverage。

# Core: 不同粒度的信息抽取任务

MIMG: 诊断工具

MIMG 不是一种训练方法, 而是一种诊断框架。通过在不同粒度上测试模型的信息抽取能力, 精确定位模型的弱点。

核心价值：如果模型在 document level 表现好但 sentence level 表现差, 说明模型的瓶颈在"精确定位"而非"大范围理解"。

LongMIT: 多轮遗忘问题

真实用户与 LLM 的交互是多轮的。第 1 轮给出长文档后, 后续多轮问答会逐渐"挤压"原始文档的注意力份额。

核心挑战：KV cache 随对话增长 -> 文档信息被对话历史"稀释" -> 模型逐渐遗忘早期上下文中的细节。
