1. 概览:一个 R1-Zero 式的编码智能体

数学与单步代码推理已经能被 RL 稳定训练(DeepScaleR 1.5B、DeepCoder 14B),但把 RL 扩到长程、多步的 agentic 任务(解 GitHub issue、写 feature、调试)仍是开放难题。DeepSWE 直接从 Qwen/Qwen3-32B 出发,只用强化学习、不做任何 SFT 或蒸馏(因此是 R1-Zero 而非 R1 式),就在最难的 SWE-Bench-Verified 上拿到开源开放权重模型的 SOTA。

# DeepSWE-Preview 关键配置与结果
base_model  = "Qwen/Qwen3-32B"   # 非 Coder 版
method      = "RL only"          # 无 SFT / 无蒸馏
framework   = "rLLM"             # agent 后训练框架
train_data  = 4_500              # R2E-Gym 任务子集
compute     = "64×H100 × 6 days"

# 数据去污: 剔除与 SWE-Bench-Verified
# 同源的仓库 (如 sympy)

# SWE-Bench-Verified (64k ctx, 100 steps)
pass_at_1   = 42.2   # 16 次运行平均
pass_at_16  = 71.0   # 理论上界
with_TTS    = 59.0   # 混合式测试时扩展 K=16

为什么值得关注

纯 RL 胜过 SFT/蒸馏:同样甚至更多训练数据、且从更强的闭源教师蒸馏的方案(R2E-Gym / OpenHands Critic / SkyWork 等),都被这套"只做 RL"的配方超过。

32B 可复现:64 张 H100 训 6 天是学术组够得着的规模,且数据/代码/日志全开源。

R1-Zero 式作者明确指出 DeepSWE 是 R1-Zero 式(直接 RL),未来还可在其上叠加 SFT 冷启动做出 R1 式版本。定位是 DeepScaleR(数学 1.5B) → DeepCoder(代码 14B) → DeepSWE(SWE 32B) 系列的长程 agent 篇。可与 davinci_env.html、swe_evolution.html 对照。

2. 环境:R2E-Gym + 四工具 + 稀疏 ORM 奖励

解一个 PR 被建模成 RL 环境:agent 拿到一个终端 + 装好代码库的文件系统,像人用 IDE 一样操作。DeepSWE 的环境包在 R2E-Gym(一个可扩展地批量构造高质量可执行 SWE 环境的 Gym)之上,动作空间是四个工具,状态就是每个工具执行后返回的 stdout/stderr。

# R2E-Gym 四工具动作空间
tools = {
  "execute_bash": # 跑命令, 回 stdout+stderr,
  "search":       # 在目录/文件里查 query,
  "file_editor":  # 看/建/替换/插入/撤销,
  "finish":       # 认为解完 → 终止轨迹,
}

# 稀疏 Outcome Reward Model (ORM)
def reward(patch):
    tests = select("Pass2Pass", "Fail2Pass")
    if run(patch, tests, timeout=5*60).all_pass:
        return 1   # 全过且未超时
    return 0       # 有一条失败 / 超时 → 0
# 官方评测限 30min, 训练为提速压到 5min

奖励的极简主义

稀疏 ORM:不给过程分,只看最终 patch 能否让选中的一组测试全绿。奖励形式化为

$r=\mathbb{1}\big[\text{all tests pass}\ \wedge\ t\lt 5\text{min}\big]$

Pass2Pass = 改动不能弄坏原本通过的测试;Fail2Pass = 必须让原本失败(暴露 bug)的测试变通过。两类都过才算真解决。

5 分钟 vs 30 分钟:训练时把测试超时从官方的 30min 压到 5min 换取吞吐,超时直接判 0。这既是加速手段,也和后面 Compact Filtering 的"超时不给正奖励"呼应。

3. Kubernetes:把 rollout 采集扩到千核

算法之外,真正卡脖子的是并行开容器。最终训练每个 RL 迭代要同时拉起 512 个 Docker 容器(batch 64 × 8 次采样),加上并行实验,任意时刻有上千容器,直接压垮 Docker 的 API server 让 dockerd 崩溃。解法是给 R2E-Gym 接上 Kubernetes,让编排器把容器调度到一个节点池上。

# 单次 RL 迭代的容器压力
containers_per_iter = 64 * 8   # BS × passes = 512
# × 并行实验 → 任意时刻 > 1000 容器
# 结果: dockerd API server 过载崩溃

# 解法: R2E-Gym 集成 Kubernetes
node = Worker(cpu=200, nvme="6TB+ local SSD")
preload(swe_bench_images)   # 层从本地盘出
                            # 免反复拉 Docker Hub

autoscaler = ClusterAutoscaler(
    scale_up_when   = "pods unschedulable",
    scale_down_when = "idle ~20 min")
# 可扩到 > 1000 CPU 核, 成本随负载走

为什么必须上 K8s

Docker 单点撑不住:RL 的高并发 + 多实验叠加,单机 daemon 是硬瓶颈。K8s 把调度分散到节点池才解得开。

本地盘预载镜像:每个 worker 200 核、6TB+ 本地 NVMe,SWE-bench 镜像预热到磁盘,几乎每层都从本地起,启动快且不被 Docker Hub 限流拖垮。

弹性伸缩:pod 短时无法调度就自动加节点;节点闲置约 20min 就回收。这样在收集数百万轨迹的同时,让算力开销与实际负载成正比——是把 32B agent RL 跑起来的工程前提。

4. GRPO++:七项改动 + Compact Filtering

多轮 agent 的 GRPO 基本做法是在 ChatML 里 mask 掉环境观测/用户消息,只对模型自己生成的 token 算 loss。DeepSWE 在此基础上把 DAPO、Dr.GRPO、LOOP/RLOO 的洞见 + 自己的两项创新揉成一个更稳、更强的 GRPO++,在 FrozenLake 等任务上验证了稳定性。

# GRPO++ = 7 项改动
1. Clip High (DAPO)      # 提高裁剪上界→促探索
2. No KL Loss (DAPO)     # 去 KL→不被 SFT 锚死
3. No Reward Std (Dr.GRPO)# 去除难度偏置
4. Length Norm (Dr.GRPO) # 除以 max ctx→去长度偏置
5. Leave-One-Out (RLOO)  # 无偏低方差 baseline
6. Compact Filtering (us)# mask 超长/超步/超时轨迹
7. No Entropy Loss (us)  # 熵损易爆炸→去掉
#   前提: base token 熵在 0.3~1 区间

# LOOP/RLOO 优势估计 (组内留一)
# A_i = R_i - mean_{j!=i} R_j

七项的逻辑

放松约束促探索:Clip High 抬高 surrogate 上界稳住熵;去 KL 让策略不被原 SFT 信赖域锁死。

去偏:去掉 reward 标准差(Dr.GRPO)消除难度偏置,让难易题目在 loss 里被正确区分;长度归一化(除以 max ctx)消除 GRPO 对错误答案越写越长的偏好。

降方差:RLOO 用组内"留一"均值做 baseline,无偏且低方差。

经验不要熵损:熵正则会让熵指数级上涨最终训练崩溃;只要 base 模型 token 熵已在 0.3~1 的健康区间,去掉熵损反而更稳。

4.1 Compact Filtering:把 overlong filtering 推广到 agent

DAPO 的 overlong filtering 只 mask 掉"打满上下文"的轨迹。多轮 agent 里终止原因更多:超时(生成太久或环境执行慢)、打满上下文、达到最大步数。DeepSWE 把这三种都 mask 出 loss,叫 Compact Filtering——只有 agent 主动 finish 提交的轨迹才参与梯度。

# Compact Filtering: 只保留主动提交
def keep_in_loss(traj):
    if traj.hit_max_context:  return False
    if traj.hit_max_steps:    return False
    if traj.timeout(20*60):   return False
    return True   # 仅"deliberate finish"计 loss

# 两大收益
# ① 防/延迟 reward collapse
#    避免"前10步对、后面乱改文件"
#    这类偶然正奖励被反复强化
# ② 每步思考↓, 跨步长程推理↑
#    avg response len ↓ 而 avg steps ↑

两个非平凡收益

防崩塌阻止 reward collapse:agent 可能"歪打正着"——碰巧改对了却继续乱动文件仍拿到正奖励,这类行为被强化并累积后会让训练崩塌。只对主动提交给奖励,逼模型认真自测、对最终提交更有把握。

思考重分配:训练中平均单步 response 长度下降、而平均环境步数上升——说明模型不再每步过度长思,而是把推理摊到更多步做长程规划。

超时口径:Compact Filtering 里生成侧超时阈值为 20 分钟(区别于 §2 里测试执行的 5 分钟奖励限时)。二者共同确保"没主动、干净地解完"的轨迹不污染梯度。

5. Test-Time Scaling:token 无效,rollout 有效

推理模型常靠扩 token 提 Pass@1(如 DeepCoder 把 ctx 从 32K→64K 让 LiveCodeBench 57.8→60.6%)。但对 SWE 任务,DeepSWE 发现扩 token 几乎没用:16K→128K 里超过 32K 后提升 ≤2%。真正有效的是扩 rollout 数量——生成 N 条轨迹,再用验证器挑出正确的那条。

# ① 扩 token: 32K 后边际递减 (≤2%)
ctx_16k_to_128k = "beyond 32K marginal"

# ② 扩 rollout: 从 N 条轨迹里选最优
trajs = [rollout(task) for _ in range(N)]

# 三种选择器
# 执行无关: DeepSWE-Verifier (训2轮)
best = verifier_llm(trajs)          # 判对错
# 执行相关: LLM 生成测试, 过得多者胜
best = argmax(t: tests_passed(t))
# 混合 (R2E-Gym): 二者结合 ← DeepSWE 用

# 结果 (SWE-Bench-Verified)
pass_at_1  = 42.2
hybrid_K16 = 59.0   # K=8 已拿大部分收益
pass_at_16 = 71.0   # 理论上界 (100% 选对)

为什么 rollout 比 token 香

SWE 是搜索问题:一次跑到底不如多试几次再挑。给定 N 条轨迹,难点变成"识别哪条真解对了"。

执行无关验证器:DeepSWE-Verifier 在正确/错误 patch 上训 2 轮,直接判轨迹对错,不跑代码。

执行相关验证器:另一个 LLM 生成多样测试与边界用例,过得最多的轨迹胜。

混合Hybrid Scaling(取自 R2E-Gym):把执行无关 + 执行相关两条路结合,显著优于任一单独验证器,让 DeepSWE 仅用 K=16 就达到 59.0%。但 Pass@K 上界(71.0)说明现有 TTS 离最优仍有很大空间。实务上 K=8 已能拿到大部分收益。

DeepSWE-Preview 在官方 R2E-Gym 代码库上评测(64K 上下文、100 最大环境步),生成的 patch 再移植回官方 SWE-bench 仓库算最终分,Pass@1 取 16 次运行平均。核心结论:仅靠 RL、在 Qwen3-32B 上就拿到 42.2% Pass@1,超过用相近/更多数据且做了蒸馏或 SFT 的先前方案。

三个值得记住的点

涌现自发的测试意识:模型学会主动补边界用例、跑仓库回归测试来验证自己的补丁;并按每步复杂度自适应分配思考 token——这些都不是显式教的,是稀疏 ORM + Compact Filtering 逼出来的。

SFT 冷启动反而更差:用 Claude-Sonnet 3.7/4 蒸馏做 SFT 冷启动、换 RL 数据/环境、以及关掉思考,都被作者试过且效果不好——印证"纯 RL 直上"这条路对 32B 是可行且更优的。

系列与路线:DeepSWE 是 DeepScaleR(数学)→DeepCoder(代码)→DeepSWE(SWE agent)的长程 agent 篇,当前为 R1-Zero 式;后续计划叠 SFT 做 R1 式、上更大模型 + 更长上下文,并推广到 web agent。全套开源便于社区在 agent RL 上继续 scaling。
