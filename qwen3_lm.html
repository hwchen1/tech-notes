设计哲学

深而窄: 94层 × 4096维，比 DeepSeek-V3 (61层×7168维) 更深但更窄。

全 MoE: decoder_sparse_step=1 意味着每一层都是 MoE，没有前几层 dense 的做法。

极致 GQA: 16:1 的 Q/KV 比，比 DeepSeek-V3 的 MLA 更简单但推理时 KV cache 更大。

No Shared Expert: 与 DeepSeek-V3 不同，Qwen3 没有 shared expert，所有知识都分布在 128 个 routed experts 中。

$\text{激活比: } \dfrac{22\text{B}}{235\text{B}} = 9.4\%$
$\text{DeepSeek-V3: } \dfrac{37\text{B}}{671\text{B}} = 5.5\%$
$\rightarrow$ Qwen3 激活比更高

Tied Embedding: 输入输出共享词表权重，节省 ~0.62B 参数。

Tied Embedding 设计选择

tie_word_embeddings=True 是 Qwen 系列一贯的选择。

优势：减少参数量、强制输入输出使用同一语义空间，有正则化效果。

代价：输入 embedding 需要兼顾"理解"和"生成"两个目标，可能互相妥协。

$\text{节省: } 151936 \times 4096 = 622\text{M}$
$\text{占比: } \dfrac{0.62\text{B}}{235\text{B}} = 0.26\%$
$\rightarrow$ 对大模型意义不大
$\rightarrow$ 更多是 Qwen 惯例

Vocab 151936: Qwen 系列从 Qwen1 开始就用这个大小，支持中英日韩多语言。比 DeepSeek 的 129280 大 17%。

3. RMSNorm & Residual Stream

# Pre-Norm 架构 (和 DeepSeek-V3 相同)

class RMSNorm:
    # weight: [4096]
    # eps: 1e-6 (Qwen3 默认)

    def forward(x):
        # x: [batch, seq, 4096]
        rms = sqrt(mean(x ** 2, dim=-1) + eps)
        return x / rms * weight

# 每层 2 个 RMSNorm:
# 1. input_layernorm (before attn)
# 2. post_attention_layernorm (before FFN)

# Residual 连接:
def layer_forward(hidden, layer):
    # Attention block
    residual = hidden
    hidden = layer.input_layernorm(hidden)
    hidden = layer.self_attn(hidden)
    hidden = residual + hidden

    # FFN block
    residual = hidden
    hidden = layer.post_attn_layernorm(hidden)
    hidden = layer.mlp(hidden)  # MoE
    hidden = residual + hidden

    return hidden

# 94层的深度带来的挑战:
# - 梯度消失/爆炸风险更高
# - RMSNorm 的稳定性至关重要
# - 可能需要 gradient scaling

# 最终层后还有一个 final RMSNorm:
# model.norm (before lm_head)

94层的设计意义

深度 vs 宽度: Qwen3 选择 94层×4096维，而非 DeepSeek-V3 的 61层×7168维。

更深的模型理论上可以学到更复杂的函数组合（compositional generalization），但训练难度更大。

$\text{Qwen3: depth}=94,\ \text{width}=4096$
$\text{DS-V3: depth}=61,\ \text{width}=7168$
$94 \times 4096 = 385\text{K}$
$61 \times 7168 = 437\text{K}$

Pre-Norm 必要性: 94层深度下，Post-Norm 几乎无法训练。Pre-Norm 让梯度路径更短（直接通过 residual stream）。

RMSNorm 参数量: 每层 2 个 + 最终 1 个 = 94×2+1 = 189 个，每个 4096 参数，总计 ~0.77M（可忽略）。

4. GQA (Grouped Query Attention)

# Qwen3 使用 GQA (非 MLA!)
# 64 Q-heads, 4 KV-heads → 16:1 比

num_attention_heads  = 64   # Q heads
num_key_value_heads  = 4    # KV heads
head_dim             = 128
attention_bias       = False # 无bias

# 投影维度:
q_proj_dim = 64 * 128  # = 8192
k_proj_dim = 4  * 128  # = 512
v_proj_dim = 4  * 128  # = 512
o_proj_dim = 64 * 128  # = 8192

# 参数量 (每层):
# Q: 4096 × 8192 = 33.6M
# K: 4096 × 512  = 2.1M
# V: 4096 × 512  = 2.1M
# O: 8192 × 4096 = 33.6M
# Total: 71.4M per layer

def gqa_attention(x, kv_cache=None):
    Q = x @ W_q  # [B, S, 8192]
    K = x @ W_k  # [B, S, 512]
    V = x @ W_v  # [B, S, 512]

    # Reshape
    Q = Q.view(B, S, 64, 128)
    K = K.view(B, S, 4, 128)
    V = V.view(B, S, 4, 128)

    # Apply RoPE to Q, K
    Q, K = apply_rope(Q, K, freqs)

    # KV repeat for GQA
    # 4 KV heads → 每个服务 16 Q heads
    K = K.repeat_interleave(16, dim=2)
    V = V.repeat_interleave(16, dim=2)

    # Scaled dot-product attention
    attn = (Q @ K.T) / sqrt(128)
    out = softmax(attn) @ V
    return out @ W_o

GQA 16:1 设计

为什么不用 MLA? GQA 更简单、更成熟、框架支持更好。MLA 需要定制 kernel，GQA 可直接用 FlashAttention。

16:1 比例: 比常见的 8:1 (Llama 3) 更激进，进一步压缩 KV cache。

$$\text{KV cache per token per layer:}$$$$= 2 \times 4 \times 128 \times 2\text{B}_{(\text{bf16})} = 2048\text{ bytes} = 2\text{KB}$$$$94\text{ layers} \times 2\text{KB} = 188\text{KB/token}$$$$32\text{K context} = 6.0\text{ GB/batch}$$

No attention bias: 去掉 bias 减少参数且不影响性能（现代共识）。

vs DeepSeek-V3 MLA: MLA 每层只需 512 元素的 latent KV，但需要额外的解压矩阵。GQA 的优势是实现简单、硬件友好。

5. MoE FFN — 128 Experts, Top-8 Softmax Router

# Qwen3 MoE 配置

num_experts           = 128
num_experts_per_tok   = 8
decoder_sparse_step   = 1  # 每层都是MoE
norm_topk_prob        = True
shared_expert_intermediate_size = 0

# Router:
class Qwen3MoERouter:
    gate: [4096, 128]  # 0.52M params

    def forward(x):
        logits = x @ gate  # [B*S, 128]
        probs = softmax(logits, dim=-1)
        top8_probs, top8_idx = topk(probs, 8)

        # norm_topk_prob=True:
        # 重新归一化 top-8 概率
        top8_probs = top8_probs / top8_probs.sum()
        return top8_idx, top8_probs

# 对比 DeepSeek-V3 的 Router:
# DS-V3: sigmoid + 分组 + aux-loss-free
# Qwen3: softmax + norm_topk + 传统 aux-loss

# 关键区别:
# 1. Softmax → 概率和为1，天然竞争
# 2. norm_topk_prob → top-8 重新归一化
# 3. 无分组 (no grouping)
# 4. 无 shared expert
# 5. 用 load balancing aux loss

# MoE 输出:
def moe_forward(x):
    idx, probs = router(x)
    out = 0
    for i in range(8):
        expert_out = experts[idx[i]](x)
        out += probs[i] * expert_out
    return out

Softmax Router 设计

Softmax vs Sigmoid: Softmax 天然互斥竞争——一个 expert 概率高则其他低。Sigmoid 允许多个 expert 同时获得高分（更灵活但需要更多 balancing）。

norm_topk_prob=True: 选出 top-8 后重新归一化，确保权重和为 1。这让每个 token 获得"完整"的表示更新。

$\text{Softmax: } \sum_{i=1}^{128} p_i = 1$
$\text{Top-8 后: } \sum_{\text{top8}} p_i < 1$
$\text{Renorm: } p_i' = \dfrac{p_i}{\sum_{\text{top8}} p_j} \Rightarrow \sum p_i' = 1$
$\rightarrow$ 不会因稀疏导致表示缩小

无 Shared Expert: 所有知识都必须通过 routing 访问。这意味着 router 需要更准确地选择 expert。

全 MoE (decoder_sparse_step=1): 每层都是 MoE，没有 dense fallback 层。比 DeepSeek-V3 (前3层 dense) 更激进。

6. Expert 结构 — SwiGLU (4096→1536→4096)

# 每个 Expert 的内部结构

moe_intermediate_size = 1536
hidden_size = 4096

class Qwen3Expert:
    gate_proj: [4096, 1536]  # 6.3M
    up_proj:   [4096, 1536]  # 6.3M
    down_proj: [1536, 4096]  # 6.3M
    # Total per expert: 18.9M

    def forward(x):
        # SwiGLU activation
        gate = swish(x @ gate_proj)
        up = x @ up_proj
        hidden = gate * up   # element-wise
        return hidden @ down_proj

# 128 experts × 18.9M = 2.42B per layer
# 94 layers × 2.42B = 227B (MoE部分)

# 激活参数 (per token per layer):
# 8 experts × 18.9M = 151M

# 对比 DeepSeek-V3 Expert:
# DS-V3: 7168 → 2048 → 7168 (44.0M/expert)
# Qwen3: 4096 → 1536 → 4096 (18.9M/expert)
# → 单个 expert 2.3× 更小
# → 但 Qwen3 选 8 个, DS-V3 也选 8 个

# Expert 粒度对比:
# DS-V3: 256 大expert (粗粒度)
# Qwen3: 128 中expert
# Mixtral: 8 超大expert (最粗)

# SwiGLU 扩展比:
# 标准 FFN: 4× (4096→16384)
# Qwen3 expert: 0.375× (4096→1536)
# → 极小的 expert，靠数量取胜

小 Expert 设计哲学

扩展比 0.375×: 传统 FFN 用 4× 扩展（4096→16384），但 Qwen3 的 expert 只有 1536 维中间层——比输入还小。

这是因为选 8 个 expert 并行，等效于：8 × 1536 = 12288 维中间层，对应 3× 扩展（接近标准）。

$$\text{等效扩展比: } \frac{8 \times 1536}{4096} = 3.0\times$$$$\text{(标准 FFN 通常 } 4\times \text{ 或 } \tfrac{8}{3}\times\text{)}$$

为什么不用 Shared Expert? Qwen 团队实验表明 128 个路由 expert 已经足够覆盖通用知识，无需额外的 always-on expert。简化实现。

参数效率: 每层 MoE 2.42B 参数但只激活 151M。参数利用率 = 151M/2.42B = 6.2%（每层）。

7. RoPE — Base 1M，原生长上下文

# Qwen3 位置编码

rope_theta = 1_000_000  # 1M (vs GPT默认10000)
max_position_embeddings = 40960
head_dim = 128  # RoPE 应用在每个 head 上

# RoPE 频率计算:
def compute_freqs(dim=128, theta=1e6):
    freqs = 1.0 / (theta ** (
        arange(0, dim, 2) / dim
    ))
    # freqs[0] = 1.0 (最高频)
    # freqs[63] = 1e-6^(126/128)
    #           = 1e-5.91 ≈ 1.23e-6
    return freqs

# 为什么 theta=1M?
# - theta=10K: 有效长度 ~4K-8K
# - theta=500K: 有效长度 ~32K
# - theta=1M: 有效长度 ~128K+
# → 不需要 YaRN 等扩展技术!

# 训练长度 vs 推理:
# max_position_embeddings = 40960
# → 训练时最长 40K tokens
# → 但 theta=1M 可外推到更长

# 对比 DeepSeek-V3:
# DS-V3: theta=10000 + YaRN(factor=40)
#   → 基础短 + 动态扩展
# Qwen3: theta=1M, 无 YaRN
#   → 原生长上下文

# RoPE 只应用在 Q 和 K 上
# V 不加位置信息 (标准做法)

# 注意: GQA 下 RoPE 应用:
# Q: [64 heads, 128 dim] 全部加 RoPE
# K: [4 heads, 128 dim] 全部加 RoPE
# → 无 nope/rope 分离 (与 MLA 不同)

θ=1M 设计选择

原生长上下文: 用大 theta 直接支持长序列，不需要 YaRN/NTK 等后处理。

代价: 短文本时低频维度几乎不旋转，浪费了部分表示能力。但 Qwen 团队认为这个 trade-off 值得。

$f_{\min} = \theta^{-126/128} = 10^6 \cdot e^{-126/128} $
$= 1.23 \times 10^{-6}$

$\text{周期} = \dfrac{2\pi}{f_{\min}} \approx 5.1\text{M tokens}$
$\rightarrow$ 理论上能区分 $>1$M 距离

max_position_embeddings=40960: 训练时只见过 40K，但 theta=1M 的外推能力可以支持更长推理。实际部署通常结合动态 NTK 扩展到 128K+。

全 head 加 RoPE: 不像 DeepSeek-V3 的 MLA 只给部分维度加 RoPE（rope_head_dim=64），Qwen3 的 GQA 给全部 128 维加 RoPE——更简单。

8. Thinking / Non-Thinking 双模态

# Qwen3 独特功能: 双模态推理
# 可以在 "thinking" 和 "non-thinking"
# 模式之间切换

# Thinking 模式 (类 o1/DeepSeek-R1):
# 输出格式:
# <think>
# [内部推理过程，用户可见]
# </think>
# [最终答案]

# Non-Thinking 模式 (标准):
# 直接输出答案，无思考过程
# 更快、token 消耗更少

# 切换方式:
# 系统 prompt 或特殊 token 控制:
# enable_thinking=True → thinking mode
# enable_thinking=False → non-thinking

# 技术实现推测:
# 1. 训练时混合两种数据:
#    - thinking: CoT 数据 + RL
#    - non-thinking: 标准 SFT
# 2. <think> token 作为模式切换信号
# 3. RL 训练 thinking 链的质量

# 与 DeepSeek-R1 对比:
# R1: 专门的思考模型 (单模态)
# Qwen3: 同一模型，双模态切换
# → 部署成本低 (一个模型两种用法)

# Thinking budget 控制:
# thinking_budget = N tokens
# → 限制思考长度
# → 平衡质量 vs 延迟

统一模型双模态

核心创新: 一个模型同时支持深度思考和快速响应，用户按需切换。不需要部署两个模型。

训练方法 (推测):

1. 先做标准 SFT (non-thinking)

2. 加入 CoT 数据 + RL (thinking)

3. 混合训练让模型学会模式切换

$\text{Thinking: 高质量，} 2\text{-}10\times \text{ tokens}$
$\text{Non-Thinking: 快速，标准输出}$

$\text{关键: 同一权重，不同行为}$
$\rightarrow$ 通过 prompt 控制

Thinking Budget: 可以限制思考 token 数量，在质量和延迟之间取舍。这比 DeepSeek-R1 的纯 thinking 模型更灵活。

vs 专用 thinking 模型: DeepSeek-R1 是专门的 reasoning 模型，不能关闭 thinking。Qwen3 更实用——一个部署覆盖两种需求。

9. 训练策略

# Qwen3 训练流程 (公开信息)

# ===== Stage 1: Pre-training =====
# 数据: ~36T tokens (Qwen3 技术报告)

# 针对 thinking mode 的强化学习
# 奖励: 答案正确性 + 推理质量
# 方法: 可能是 GRPO 或类似

# ===== Stage 4: Mode Fusion =====
# 融合 thinking 和 non-thinking 能力
# 确保两种模式都保持质量
# 可能用 DPO 或 多任务 RL

# 训练基础设施 (推测):
# - 数千 H100/A100 GPU
# - Expert Parallelism + TP + PP
# - BF16 或混合精度

# MoE 训练特殊考虑:
# - Load balancing loss (标准)
# - Expert 激活均衡性
# - Communication overhead
# - 128 experts → 高 all-to-all 成本

# 全 MoE (94层) 的训练挑战:
# - 无 dense 层做 "锚点"
# - 更容易出现路由坍塌
# - 需要精心设计 balancing 策略

训练亮点

36T tokens: Qwen3 大幅增加了预训练数据量（Qwen2 是 18T），更多数据对大 MoE 模型的路由多样性至关重要。

全 MoE 训练: 94层全部是 MoE，训练时需要维护 128×94=12032 个 expert 的负载均衡。

$\text{Expert 总数: } 128 \times 94 = 12{,}032$
$\text{每层 All-to-All 通信:}$
$128 \text{ experts 分布在 } N \text{ 节点}$
$\text{通信量} = O(\text{batch} \times \text{seq} \times d)$
$\rightarrow$ 主要瓶颈是网络带宽

Mode Fusion 是关键: 如何在同一个模型中平衡 "快速回答" 和 "深度思考" 是 Qwen3 最独特的训练挑战。

Load Balancing: 使用标准 aux loss（和 DeepSeek-V3 的 aux-loss-free 不同），需要调超参防止影响模型质量。

10. 推理优化

# Qwen3-235B-A22B 推理特性

# ===== KV Cache =====
# GQA 4 heads × 128 dim × 2(K,V)
# = 1024 元素/层/token
# × 94 层 = 96256 元素/token
# × 2 bytes (BF16) = 192.5 KB/token
# 32K context = 6.0 GB/batch

# 对比 DS-V3 MLA:
# 512 dim × 61 层 = 31232 元素/token
# = 61 KB/token → 3× 更小!

# ===== 激活参数 =====
# 每 token 激活 22B 参数
# 需要将 22B 参数加载到 HBM
# → 内存带宽瓶颈

# ===== Expert Offloading =====
# 235B 参数 BF16 需要 ~470GB
# 8× H100 (80GB each) = 640GB
# 刚好放得下
# 或: 4× H100 + CPU offload

# ===== MoE 推理优化 =====
# 1. Expert Parallelism
#    128 experts / N GPUs
# 2. Expert Caching
#    热门 expert 常驻 HBM
# 3. Prefetching
#    预加载下一层可能用的 expert

# ===== 量化 =====
# INT4/INT8 量化 → 4-8× 压缩
# GPTQ, AWQ, GGUF 都支持
# INT4: ~60GB, 单卡 A100 可跑

推理效率分析

22B 激活: 每 token 只激活 22B 参数，FLOPS 仅相当于 22B dense 模型——这是 MoE 的核心优势。

KV Cache 较大: GQA 4 heads 的 KV cache 比 DeepSeek-V3 的 MLA 大 3×。这是 Qwen3 的弱势。

$\text{单 token FLOPS} \approx 2 \times 22\text{B} = 44\text{ GFLOPS}$

$\text{vs Dense 235B:}$
$\approx 2 \times 235\text{B} = 470\text{ GFLOPS}$

$\text{MoE 加速比: } \dfrac{470}{44} \approx 10.7\times$

Expert Parallelism: 128 experts 可以分布在多 GPU 上。每 GPU 16 experts（8 GPU），每 token 需要 all-to-all 通信选择 8 个。

Thinking 模式开销: thinking mode 会生成额外的推理 token (2-10×)，增加延迟和成本。需要在部署时做好 budget 控制。
