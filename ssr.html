Self-play SWE-RL (SSR) — 自博弈训练超级软件智能体

Meta / CWM 团队 · arXiv 2512.18552 | 单一 LLM 策略同时扮演"造 bug 的挑战者"与"修 bug 的求解者"两个角色，共享参数、联合 RL 自我进化 · 极简数据假设(只要装好依赖的 Docker 仓库镜像，无需人类 issue / 现成测试 / 测试框架先验) · bug 用"测试套件的改进"形式化定义而非自然语言 · 基座 CWM-sft(32B) · SWE-bench Verified +10.4 / SWE-Bench Pro +7.8，全程超越人类数据基线，并泛化到没训过的自然语言 bug 描述

1. 概览：从"喂人类数据"到"自博弈自进化"

训练强 SWE 智能体的主流做法依赖人类策划的数据——GitHub issue 描述、现成的 fail-to-pass 测试、评测脚本。这类数据既稀缺又与被训模型脱节：静态的 bug 生成流水线不会随模型变强而变难，因此系统无法持续自我提升。SSR 的核心思路借鉴 AlphaZero——只给"游戏规则"（一个能跑测试的沙盒仓库），让同一个 LLM 策略在造 bug（挑战者）↔ 修 bug（求解者）的循环里对抗式自我进化。与"零数据"自博弈(Absolute Zero / R-Zero)不同，SSR 把博弈接地在真实代码仓库上(类似 SPICE 的语料接地)，从而能获得超出模型已有知识的新技能。

# 同一套策略权重, 靠 prompt 切换两个角色
policy = load("CWM-sft-32B")   # RL 之前的 sft 检查点

for step in range(150):          # ~2.5B tokens
    # 角色A: 挑战者 = 造 bug
    artifact = policy.inject_bug(sandbox_repo)
    ok, s   = validate(artifact)   # 一致性 + 求解率 s

    # 角色B: 求解者 = 修 bug
    patch   = policy.repair(buggy_repo)
    passed  = run_tests(patch)

    # 两个奖励信号 联合更新同一策略
    policy.rl_update(r_inject(ok, s),
                     r_solve(passed))

为什么是自博弈

范式数据脱节问题：人类 issue 高质量但稀缺，且静态数据不会随模型变强而升级难度，学习信号很快枯竭。

AlphaZero 类比：只给规则、靠自博弈探索规则的蕴含，就能到超人水平。SSR 把"规则"设为"沙盒里能跑测试的仓库"。

接地(grounding)：纯"零数据"自博弈学不到环境规则之外的知识；SSR 让挑战者在真实仓库上造 bug，从真实代码分布里源源不断汲取新挑战。

一句话：同一个模型既当出题人又当答题人，共享权重、联合更新——出题能力和解题能力互相拉扯，自动生成一条"永远卡在能力前沿"的课程。

2. Agentic Bug Injection：bug 五件套与一致性校验

SSR 的极简输入假设是全篇最狠的一点：只假设有一批"装好依赖、含源码的 Docker 镜像"，不假设有测试解析器、现成测试、跑测试的命令，甚至不知道是什么语言/测试框架。挑战者必须自己通过与环境交互去发现怎么跑测试、怎么解析结果。一个 bug 被形式化为一个五件套 artifact——它同时能验证 bug 存在、验证修复、解析测试、定义 bug、以及"藏"住 bug。

# bug artifact = 5 个文件
test_script.sh    # 跑测试套件的 bash
test_files.txt    # oracle 测试文件清单;
                  # 评测前强制重置回原版防作弊
test_parser.py    # 解析输出 → {test_id: pass/fail}
                  # 语言无关, 用 Python 图省事
bug_inject.diff   # 注入 bug 的 git diff
test_weaken.diff  # 删/弱化测试来"藏" bug;
                  # 其"反转"即为给求解者的规格

# 输入假设 (极简)
assert have(docker_image_with_src_and_deps)
assert not have(tests, parser, run_cmd, lang_prior)

五件套的巧思

形式化bug = 测试改进而非自然语言：把 test_weaken.diff（弱化测试藏 bug）反转过来，就得到"期望行为的规格"，直接当求解者的任务说明——完全绕开难以自动评估的自然语言 issue。

评测完整性：test_files.txt 里的 oracle 测试在评测前强制重置回原版，即使 agent 去改测试"糊弄"，评测时也用原始测试，杜绝作弊。

逆向变异测试(inverse mutation testing)：把每个被改文件单独还原，若能让某失败测试转绿，则该文件对触发 bug 必要——反过来做经典变异测试，确保 bug 补丁没有冗余/无关改动。

3. 复杂 bug 生成与"高阶 bug"

好的挑战者要能造出多样、逼近真实开发复杂度的 bug。SSR 用两个简单策略：(1) 删除代码文件/hunk，(2) 借 git log 洞察选择性回退历史改动——两者都逼求解者去新增/修改代码（现实软件演化里最高频的改动类型）。为保证是真语义错误而非语法崩，要求 agent 做兼容性修补让项目仍可运行。更关键的是高阶 bug：把求解者失败的修复尝试作为新的 buggy 状态，重新封装成 bug artifact——这既模拟了开发者在自己代码上无意引入 bug 的真实过程，也让 bug 生成可持续扩展（长期运行不会枯竭为重复模式）。

# 两种一阶 bug 生成策略
def inject(repo):
    if coin():                 # 策略1
        remove_hunks_or_files(repo)     # 大段删除
    else:                       # 策略2 (history)
        revert_git_history(repo)        # 回退历史改动
    fix_compat()               # 保持可运行→真语义 bug

# 质量控制参数
require(min_changed_files, min_passing_tests,
        min_failing_tests)

# 高阶 bug: 求解者失败尝试 → 新 bug
if not solver_attempt.passed:
    higher_order = wrap_as_artifact(
        base + bug_inject + test_weaken
             + "pred_patch.diff")   # 失败补丁
    # 只做到二阶, 防与已有 bug 重叠

难度与多样性来源

消融直接注入最差：朴素 prompt 只会造出 var=0→var=1 这种一行改动，信号太弱；删除式逼求解者重建缺失功能、理解仓库结构，明显更强。

高阶 bug把"求解者自己搞砸的中间状态"回收成新任务，覆盖模型自身编码能力的各个盲区，也解决了纯删除法"从零重建、不够自然"的问题。

可扩展性(scalable)：只做到二阶(不再往上)以降低与已有 bug 重叠的概率。"removal + history"随机混采两个 prompt，实验中综合最好。

4. Agentic Bug Repair 与对抗式奖励设计

bug 通过一致性校验后，求解者从构造好的 buggy 代码库出发修复。构造过程：原库 → 打 bug_inject.diff → 打 test_weaken.diff（高阶再叠 pred_patch.diff）→ 删掉 .git 重新初始化防止从 git 历史泄漏答案。任务说明就是反转的 test_weaken 补丁（不合成自然语言 issue，因其质量难自动评估）。求解者奖励是简单二元的；挑战者奖励则精心设计，用求解率 $s$ 把 bug 难度拉到"求解者能力前沿"——这构成一对对抗性激励。

# 求解者奖励: 简单二元
r_solve = +1 if all_tests_pass else -1

# 挑战者奖励 (s = 求解率 ∈ [0,1])
# α=0.8, 惩罚"退化难度"但仍给梯度
$r_{\text{inject}}=\begin{cases}-1.0 & \text{一致性校验失败}\\ -\alpha & s=0 \ \text{或}\ s=1\\ 1-(1+\alpha)s & 0 \lt s \lt 1\end{cases}$
# 对抗性激励 (Appendix B)
# 求解者期望奖励:
$\mathbb{E}[r_{\text{solve}}]=2s-1 \uparrow s$
$r_{\text{inject}}=1-(1+\alpha)s \downarrow s$
# 求解者要 s↑(易), 挑战者要 s↓(难)
# → 把 bug 推到能力前沿

奖励的关键设计

对抗相反激励：求解者期望奖励 $2s-1$ 随 $s$ 增(想要简单)，挑战者 $1-(1+\alpha)s$ 随 $s$ 减(想要难)——两者把 bug 推向刚好在求解者能力前沿。

不惩罚到死：$s=0$/$s=1$ 的退化 bug 只罚 $-\alpha$($\alpha{=}0.8$)而非 $-1$，让挑战者仍有信息梯度去校准难度。

防泄漏：重建 .git 断开历史；oracle 测试评测时重置——两道防线堵住 reward hacking。

为何不用自然语言 issue：自动评估自由文本质量太难，且高质量 NL issue 稀缺。下游 issue-solving 的提升纯粹来自"学会写通过测试的代码"，而非同分布泛化——这正解释了它能泛化到没训过的 NL bug 描述。

5. 实验结果与消融

基座用 CWM-sft(32B)（RL 之前的检查点，公平对比不同学习策略）。评测 SWE-bench Verified(500) 与 SWE-Bench Pro 公开集(731)，单次尝试、temp=1.0/top-p=0.95。关键对照是 baseline RL vs SSR——两者训在同一批环境镜像上，唯一差别是能拿到什么信息：baseline RL 能看 NL issue 描述、pass/fail 测试、评测脚本；SSR 只有裸镜像，全靠自博弈发现问题、造解、验证。训练 150 步(~2.5B tokens)，512×H100(64 训练 + 448 rollout)。

核心发现

主结果无数据也能自进化：SSR 全程未用任何任务专属训练数据，仅靠与裸仓库交互就稳定自我提升(+10.4 / +7.8)。

超越人类数据基线：在两个基准全程都压过能看 issue+测试的 baseline RL——自生成任务比人工数据提供了更丰富有效的学习信号。

泛化：训练时求解者看的是"反转测试补丁"这种形式化规格，从没见过自然语言 issue；但因为学到的是"写通过测试的代码"这一底层能力，故能迁移到 SWE-bench 那种 NL 描述的真实任务。

6. 挑战者-求解者博弈分析与洞察

附录 A 从理论上剖析了这个自博弈的隐患：与围棋象棋那种"充分自博弈→超人"的两人零和游戏不同，一个足够聪明的挑战者存在"支配策略"，能让求解者学不到任何有用东西、自博弈就此停滞。这解释了论文一个刻意的设计选择——不追求深度充分的自博弈，而是靠强接地 + 有限自博弈来规避退化。

# 挑战者有唯一最优目标求解率 p*

$p^{*}\approx \dfrac{a}{a+b}\in(0,1),\quad p^{*}\!\approx 0.2$

博弈的隐患与对策

理论最优目标求解率 $p^{*}\approx0.2$：奖励虽在 $s\to0$ 递增却在 $s{=}0$ 为负，加上 $G{=}8$ 采样噪声，逼出一个 $(0,1)$ 内的唯一峰值——挑战者的"理性"目标就是让 bug 约 20% 可解。

支配挑战者：若能改测试(如 hash(code) 伪随机 + 混淆代码库)，可无视求解者稳拿 $p^{*}$，求解者最优反应是"什么都不做"——自博弈死锁。

隧道视野：即便动作受限，只调单一难度旋钮(混淆强度/任务链长度)也会停滞、丧失多样性(Absolute Zero 同样有此风险)。

三条缓解(部分已采用)：① 挑战者接地在大而多样的真实仓库；② 别让挑战者偏离初始指令式策略、避免陷入隧道视野；③ 别指望用无人类接地的自博弈提升自然语言——技术/代码世界模型知识可以，但与人交流的 NL 会漂移(类比 Diplomacy / 最后通牒博弈)。

总评与局限：SSR 是"用极简假设 + 自博弈训练软件智能体"的第一步——只要沙盒仓库就能自造自解、难度自升，无需人类 issue/测试。局限：结果尚属早期；理论上挑战者的支配策略要求刻意限制自博弈深度；自然语言沟通不宜靠纯自博弈提升。可与 davinci_env.html(环境合成/难度筛选)、agentfounder.html、mirothinker.html 等 agent 训练笔记对照——daVinci 靠大规模人类 PR 合成 + 难度筛选喂 SFT，SSR 则走零人类数据 + 自博弈 RL的另一条路。
