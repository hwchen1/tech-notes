KV Cache 深度解析

从训练到推理 — 理解 Key-Value 缓存的本质、优化与前沿技术

Part 1: 推理时的 KV Cache

1. KV Cache 基本原理

自回归解码中，每个新 token 需要 attend 到所有历史 token。不使用缓存时每步重算所有 K,V，总复杂度 O(N²)；使用 KV Cache 后每步仅计算新 token 的 K,V，复杂度降为 O(N)/step。

# Cached attention forward
def attention_with_cache(x, cache_k, cache_v, W_q, W_k, W_v, W_o):
    # x: [1, d] — single new token
    q = x @ W_q  # [1, d]
    k = x @ W_k  # [1, d]
    v = x @ W_v  # [1, d]

    # Append to cache
    cache_k = cat([cache_k, k], dim=0)  # [seq+1, d]
    cache_v = cat([cache_v, v], dim=0)  # [seq+1, d]

    # Attend to ALL cached keys
    attn = softmax(q @ cache_k.T / sqrt(d))
    out = attn @ cache_v  # [1, d]
    return out @ W_o, cache_k, cache_v

KV Cache 将 O(N²) 的重复计算降为 O(N)，代价是线性增长的显存占用

核心思想：空间换时间。缓存历史 token 的 K、V 向量，避免每步重复投影计算。

无缓存：生成第 N 个 token 时需重新计算前 N-1 个 token 的 K,V，总计算量 O(1+2+...+N) = O(N²)

有缓存：每步只计算 1 个新 token 的 K,V 并追加到缓存，总计算量 O(N)

2. KV Cache 内存分析

KV Cache 的显存占用与模型规模、序列长度、batch size 成正比，在长序列推理中往往超过模型权重本身。

def kv_cache_memory_bytes(
    num_layers: int,
    num_kv_heads: int,
    head_dim: int,
    seq_len: int,
    batch_size: int = 1,
    dtype_bytes: int = 2,  # FP16
) -> int:
    # 2 = K + V
    return (
        2 * num_layers * num_kv_heads
        * head_dim * seq_len
        * batch_size * dtype_bytes
    )

# LLaMA-70B example
mem = kv_cache_memory_bytes(
    num_layers=80,
    num_kv_heads=8,   # GQA: 8 KV heads
    head_dim=128,
    seq_len=4096,
)
# = 2*80*8*128*4096*2 = 1.07 GB/request
# batch=8 → 8.6 GB KV cache alone

KV Cache 随 batch_size x seq_len 线性增长，长序列场景下可超过模型权重

公式：
Memory = 2 x L x H_kv x d x S x B x dtype

关键变量：
- L: 层数 (80 for 70B)
- H_kv: KV head 数
- d: head dimension
- S: 序列长度
- B: batch size

实际影响：在 serving 场景中，KV Cache 决定了单 GPU 能同时服务多少请求（throughput vs latency tradeoff）

3. GQA/MQA 对 KV Cache 的压缩

通过减少 KV head 数量来压缩缓存：MHA 全量缓存，MQA 极致压缩（1个KV head），GQA 在两者间取平衡。

def gqa_attention_cached(
    x, cache_k, cache_v,
    W_q, W_k, W_v, W_o,
    num_q_heads=32,
    num_kv_heads=8,  # GQA group=4
):
    q = x @ W_q  # [1, num_q_heads, d]
    k = x @ W_k  # [1, num_kv_heads, d]
    v = x @ W_v  # [1, num_kv_heads, d]

    cache_k = cat([cache_k, k], dim=1)
    cache_v = cat([cache_v, v], dim=1)

    # Repeat KV heads to match Q heads
    group = num_q_heads // num_kv_heads
    k_exp = cache_k.repeat_interleave(group, dim=0)
    v_exp = cache_v.repeat_interleave(group, dim=0)

    attn = softmax(q @ k_exp.T / sqrt(d))
    out = attn @ v_exp
    return out @ W_o, cache_k, cache_v

GQA 在 MHA 和 MQA 之间取得平衡：LLaMA-2/3 使用 GQA (8 KV heads)

MHA (Multi-Head):
每个 Q head 有独立的 K,V head
Cache = num_q_heads 份

GQA (Grouped-Query):
多个 Q head 共享一组 K,V
Cache 压缩 group_size 倍
LLaMA-70B: 64Q / 8KV = 8x 压缩

MQA (Multi-Query):
所有 Q head 共享 1 组 K,V
最大压缩但质量有损
适合 code generation 等场景

4. MLA (Multi-Latent Attention) KV Cache 压缩

DeepSeek-V2/V3 和 Kimi-K2 采用的低秩压缩方案：将 KV 投影到低维 latent 空间缓存，推理时再解压回原始维度。

class MLACachedInference:
    def __init__(self, d=128*128, kv_lora_rank=512):
        # Down-project KV to latent
        self.kv_down = Linear(d, kv_lora_rank)
        # Up-project latent back to KV
        self.k_up = Linear(kv_lora_rank, d)
        self.v_up = Linear(kv_lora_rank, d)

    def forward(self, x, latent_cache):
        # Compress KV into latent
        c = self.kv_down(x)  # [1, kv_lora_rank]

        # Cache the COMPRESSED latent only!
        latent_cache = cat([latent_cache, c], dim=0)

        # Decompress for attention
        k = self.k_up(latent_cache)  # [seq, d]
        v = self.v_up(latent_cache)  # [seq, d]

        # Standard attention with decompressed KV
        attn = softmax(q @ k.T / sqrt(d))
        return attn @ v, latent_cache
    # Compression: 512 / (2*128*128) = 1.5%!

MLA 将 KV 压缩到低秩 latent 空间，缓存量减少 ~98%

核心思想：
KV 信息存在大量冗余，可用低秩矩阵近似。

压缩比：
kv_lora_rank / (2 x n_heads x head_dim)
= 512 / (2 x 128 x 128) = 1.5%

代价：
推理时需额外做 up_proj 解压计算，但远小于重新计算 KV 的开销。

采用模型：
DeepSeek-V2, DeepSeek-V3, Kimi-K2

5. PagedAttention (vLLM)

借鉴操作系统虚拟内存的分页机制，将 KV Cache 按固定大小的 page 管理，消除预分配浪费，支持请求间内存共享。

# Simplified PagedAttention logic
class PagedKVCache:
    def __init__(self, page_size=16, num_pages=1024):
        self.page_size = page_size
        # Physical page pool on GPU
        self.pool = zeros(num_pages, page_size, d)
        self.free_pages = list(range(num_pages))

    def allocate_page(self):
        return self.free_pages.pop()

    def append_token(self, seq_id, k, v):
        page_table = self.page_tables[seq_id]
        slot = self.seq_lens[seq_id] % self.page_size
        if slot == 0:  # Need new page
            page_table.append(self.allocate_page())
        page_idx = page_table[-1]
        self.pool[page_idx, slot] = cat([k, v])
        self.seq_lens[seq_id] += 1

    def get_kv(self, seq_id):
        # Gather from scattered pages
        pages = self.page_tables[seq_id]
        return cat([self.pool[p] for p in pages])

PagedAttention 将 KV Cache 利用率从 ~60% 提升到 >95%

问题：
传统方案预分配 max_seq_len 的连续内存，短序列浪费大量显存。

方案：
像 OS 管理虚拟内存一样管理 KV Cache：
- 固定大小 page (如16 tokens)
- Page table 映射逻辑→物理
- 按需分配，用完释放

效果：
vLLM 通过 PagedAttention 将 serving throughput 提升 2-4x（相同 GPU 内存下服务更多并发请求）

Part 2: 训练时的 KV Cache

6. 训练时为什么通常不用 KV Cache

训练时整个序列并行处理，Q/K/V 同时计算，FlashAttention 融合了整个注意力计算，无需缓存中间 K,V。

# Training: full sequence, no cache
def train_attention(x, W_q, W_k, W_v, W_o):
    # x: [batch, seq_len, d] — full sequence
    Q = x @ W_q  # [B, N, d] all at once
    K = x @ W_k  # [B, N, d] all at once
    V = x @ W_v  # [B, N, d] all at once

    # FlashAttention: fused, no N×N matrix
    out = flash_attn(Q, K, V, causal=True)
    return out @ W_o

# Inference: sequential, with cache
def infer_attention(x_new, cache_k, cache_v, ...):
    # x_new: [1, 1, d] — one token at a time
    q = x_new @ W_q  # [1, 1, d]
    k = x_new @ W_k  # [1, 1, d]
    v = x_new @ W_v  # [1, 1, d]
    cache_k = cat([cache_k, k])
    cache_v = cat([cache_v, v])
    out = attn(q, cache_k, cache_v)
    return out @ W_o, cache_k, cache_v

训练时全序列并行处理，无需逐 token 生成，KV Cache 无用武之地

训练 vs 推理的本质区别：

训练：
- 输入完整序列 [N tokens]
- Q,K,V 一次性计算
- FlashAttention 融合计算
- 需要反向传播梯度
- 无需缓存

推理：
- 逐 token 自回归生成
- 每步只有 1 个新 token
- 需要历史 K,V 做 attention
- 缓存避免重复计算
- KV Cache 是必需的

7. 训练中使用 KV Cache 的场景

虽然标准训练不需要 KV Cache，但某些特殊训练场景可以利用缓存来加速或节省显存。

# Prefix KV Cache in SFT training
# Shared system prompt cached across samples
class PrefixCachedSFT:
    def __init__(self, model, system_prompt):
        # Pre-compute KV for shared prefix
        with torch.no_grad():
            self.prefix_kv = model.compute_kv(
                system_prompt  # same for all samples
            )  # [layers, 2, prefix_len, d]

    def train_step(self, user_input, response):
        # Reuse cached prefix KV
        full_kv = cat([
            self.prefix_kv,  # frozen, no grad
            model.compute_kv(user_input),  # grad
        ])
        # Only compute loss on response tokens
        logits = model.decode(
            response, past_kv=full_kv
        )
        loss = cross_entropy(logits, response)
        loss.backward()  # grad only for user+resp

# Saves: prefix_len / total_len compute per sample
# E.g., 2K system prompt, 4K total → 50% savings

SFT 中共享 system prompt 的 KV 可跨样本复用，节省 30-50% 前向计算

场景 1: Prefix Caching (SFT)
共享 system prompt 的 KV 预计算并冻结，跨 batch 复用。

场景 2: Speculative Decoding
Teacher 模型缓存 KV 用于验证 student 的 draft tokens。

场景 3: Context Distillation
长上下文 KV 缓存后，student 从中学习压缩表示。

场景 4: Inference-aware Training
训练时模拟推理行为（带 cache），使模型适应推理时的数值特性。

8. KV Cache 量化与稀疏

通过量化、滑动窗口、稀疏保留等策略进一步压缩 KV Cache，在有限显存下支持更长序列。

# Sliding Window + Attention Sink
class StreamingLLMCache:
    def __init__(self, sink_size=4, window=1024):
        self.sink_size = sink_size  # keep first N
        self.window = window       # sliding window

    def update_cache(self, cache_k, cache_v, new_k, new_v):
        cache_k = cat([cache_k, new_k], dim=1)
        cache_v = cat([cache_v, new_v], dim=1)
        seq_len = cache_k.shape[1]

        if seq_len > self.sink_size + self.window:
            # Keep: [sink tokens] + [recent window]
            sink_k = cache_k[:, :self.sink_size]
            sink_v = cache_v[:, :self.sink_size]
            win_k = cache_k[:, -self.window:]
            win_v = cache_v[:, -self.window:]
            cache_k = cat([sink_k, win_k], dim=1)
            cache_v = cat([sink_v, win_v], dim=1)

        return cache_k, cache_v
    # Fixed memory: sink_size + window tokens
    # Supports infinite generation length!

不同策略在 memory-quality tradeoff 上的定位不同，可组合使用

Sliding Window (Mistral):
只保留最近 W 个 token 的 KV。固定内存，但丢失远距离依赖。

H2O (Heavy Hitter Oracle):
动态保留 attention score 高的 token（"heavy hitters"），淘汰低分 token。

StreamingLLM:
Attention Sink (前几个 token) + Sliding Window。发现前几个 token 即使语义无关也承载大量 attention。

KV Quantization:
将缓存从 FP16 量化到 INT8/INT4，正交于上述策略，可叠加使用。
组合策略：实际部署中常组合使用：GQA (结构压缩) + PagedAttention (内存管理) + KV Quantization (数值压缩) + Sliding Window (长度限制)，可将 KV Cache 开销降低 10-50x。
