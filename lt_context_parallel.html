Ring Attention: 4 GPU 环形 KV 传递 + Compute-Comm Overlap

CP 核心思想

问题: 序列太长（128K-1M tokens），单卡放不下完整 KV cache。需要把序列切分到多个 GPU 并行处理。

切分方式: 长度 N 的序列切成 P 个 chunk，GPU_i 持有第 i 个 chunk 的 Q/K/V（各 N/P tokens）。

Ring Attention 算法

Q 留本地: 每个 GPU 的 Q 不动。K/V 沿环形拓扑传递。经过 P 步后，每个 GPU 见过所有 KV。

Online Softmax: 不需要等全部 KV 到齐才算 softmax。每收到一个 KV chunk 就增量更新 output — 用 max/sum 统计量做 rescale。

Compute-Comm Overlap: 计算 step i 时，step i+1 的 KV 已经在传输。只要 compute ≥ comm 时间，通信完全隐藏。

因果 Mask 优化

关键: Causal attention 中，Q_i 只看 K_j (j≤i)。来自"未来"的 KV chunk 可以直接跳过不计算 → 有效步数从 P 降到 ~P/2。

通信模式对比

模式特点适用
Ring逐步传 KV，P 步完成seq 长、head 少
Ulysses按 head 切 → all-to-allhead 多、节点内
USP (混合)Ulysses 节点内 + Ring 节点间大规模集群
核心约束: CP 的效率取决于"每个 Q chunk 需要看多少 KV"。Full Attention 必须看全部 → 通信 P 步；SWA 只看窗口内 → 通信 1-2 步；Linear Attention 不做 Q×K → 没有 KV 传输需求，但有状态依赖。

2. 三种 Attention 的 CP 难点 — 从通信模式到串行依赖

# ═══ Case A: Full Attention CP ═══
# 每个 Q chunk 必须看所有 KV → P 步 ring

# 通信量 (per GPU per step):
# 2 × (N/P) × d × sizeof(bf16) bytes
# 总通信: (P-1) × 上面 ≈ 2Nd bytes

# 难点 A1: Compute-Comm Overlap
# 计算: (N/P)² × d flops per step
# 通信: 2(N/P)d bytes per step
# ratio = (N/P)d / (2×bandwidth)
# → chunk 越大越容易 overlap
# → CP 太大 (P太大) → chunk 太小 → 无法隐藏通信

# 难点 A2: Causal Mask 导致 Load Imbalance
# GPU_0 的 Q₀ 只看 K₀ (1 step)
# GPU_P-1 的 Q_{P-1} 看所有 K (P steps)
# → 前面的 GPU 空转等后面
# 解法: Zigzag 分配
# GPU_i 持有 chunk i 和 chunk 2P-1-i

# ═══ Case B: SWA (Sliding Window) CP ═══
# 窗口 w，chunk_size = N/P
# Q_i 只需要 K_j where |i-j| × (N/P) ≤ w

# 关键区别: 大部分 chunk 对不交互!
num_comm_steps = ceil(w / (N/P))  # 通常 1-2 步
# vs Full Attention 的 P 步!

# 难点 B1: 窗口跨 chunk 边界
# chunk 边界处的 token 需要相邻 chunk 的 KV
# 需要精确计算哪些 KV 要传
# → 不再是简单的"传整个 chunk"
# → 需要 partial KV 传输或 halo exchange

# 难点 B2: 不规则通信 pattern
# Ring attention 假设每步传相同量
# SWA: 只和相邻 1-2 GPU 通信
# → Ring 模式浪费，改用 P2P send/recv
# → 或者用 "shifted window" 避免边界

# 难点 B3: 与 Global Attention 层交替
# Gemma3: 5层 SWA + 1层 Global
# Global 层仍需 full ring → 两种 CP 策略共存

# ═══ Case C: Linear Attention / SSM CP ═══
# S_t = f(S_{t-1}, k_t, v_t)  ← 串行!

# 根本问题: 不是 Q×K 的 pattern
# 没有"KV 传递"这个概念
# 而是 STATE 依赖: chunk 2 需要 chunk 1 的
# 最终状态 S_{N/P} 才能开始计算

# 难点 C1: 不可消除的串行依赖
# GPU_1 必须等 GPU_0 算完 state
# GPU_2 必须等 GPU_1 算完 state
# → 有 P-1 步串行等待 (pipeline bubble)

# 解法: Parallel Prefix Scan
# 如果 f 是结合律运算:
# S_t = A_t · S_{t-1} + B_t
# 可以用 O(log P) depth 的 parallel scan
# 但需要 materialize 中间 state → 内存大

# 难点 C2: State 大小 vs KV 大小
# Linear Attn state: d×d (如 128×128 = 16K)
# 传 state 很小 → 通信不是瓶颈
# 瓶颈是串行计算等待!

# 难点 C3: 非结合律运算
# DeltaNet: S_t = S - β(S·k)k + kv
# Titans: gradient-based update
# → 不满足结合律 → parallel scan 不适用
# → 只能串行或近似

三种 CP 模式对比: Full(全连接) / SWA(仅相邻) / Linear(串行传state)

A: Full Attention CP

通信模式: 每个 Q chunk 需要全部 KV → ring 传 P-1 步。通信量大但 pattern 完全规则。

难点 1 — Overlap: chunk_size = N/P。P 太大 → chunk 太小 → compute time \lt comm time → 通信暴露。实践中 CP≤8 比较安全。

难点 2 — Load Imbalance: Causal mask 下前面的 chunk 计算量少、后面的多。解法: Zigzag 分配（GPU_i 同时持有最前和最后的 chunk）。

成熟度: 最高。Megatron-Core、DeepSpeed、vLLM 都有生产实现。

B: SWA CP

通信模式: 只和持有"窗口范围内 KV"的相邻 GPU 通信。大部分 chunk 对零交互。

难点 1 — Halo Exchange: 窗口跨 chunk 边界时，需要从相邻 GPU 取部分 KV（不是整个 chunk）。实现比简单 ring shift 复杂。

难点 2 — 混合层: Gemma 3 每 6 层有 1 层 Global Attention。该层需要 full ring CP，其余 SWA 层只需相邻通信 → 同一模型两种 CP 策略交替。

优势: 通信量从 O(N) 降到 O(w)，CP 效率反而比 Full Attention 更高。

C: Linear Attention / SSM CP

通信模式: 不传 KV，传 state（d×d 矩阵，很小）。通信本身不是瓶颈。

根本难点 — 串行依赖: \(S_t = f(S_{t-1}, x_t)\)。chunk 2 需要 chunk 1 的最终 state 才能开始。P 个 GPU 存在 P-1 步串行等待。

解法: 若 f 满足结合律（如 \(S_t = A_t S_{t-1} + B_t\)），可用 parallel prefix scan → O(log P) depth。Mamba-2、GLA 满足；DeltaNet、Titans 不满足。

不满足结合律时: 只能串行传 state，或用近似方法（如 reset state at chunk boundary → 牺牲质量换并行）。

本质区别: Attention 的 CP 是数据并行问题（同一计算对不同数据）→ 天然可并行。SSM 的 CP 是前缀计算问题（后一步依赖前一步结果）→ 本质串行，只能用 scan 降到 log 深度。

CP=8 通信量/串行深度对比 + 混合架构层模式

混合架构的 CP 设计哲学

核心洞察: 不是每层都需要跨 GPU 通信。SSM/Mamba 层的计算是 O(N) 本地的 — 每个 GPU 独立处理自己的 chunk，只需在层末传递一个小 state。把"昂贵的 ring communication"留给少数 Attention 层。

三种混合策略

Nemotron Ultra: ~80% Mamba 层（本地 + scan）+ ~20% Attention 层（ring CP）。通信量比纯 Transformer 降 5×。

Gemma 3: 5/6 SWA 层（只传 halo）+ 1/6 Global 层（full ring）。通信量比纯 Transformer 降 ~5×。

Jamba: 7/8 Mamba 层 + 1/8 Attention 层。通信降 ~7×。

选型指南

CP ≤ 4: 纯 Transformer 可行，overlap 住通信。

CP 8-16: 混合架构优势显现。Mamba/SWA 层的通信几乎为零。

CP ≥ 32: 纯 Transformer 通信占比太高。必须用 SWA 或 SSM 降通信。这是 1M context 模型都用混合架构的根本原因。

终极结论: CP 的瓶颈不是"哪种 attention 最难做 CP"，而是"哪种架构在 CP 下总效率最高"。答案是混合: 用少量 Attention 保精确 recall（承担 ring 通信），用大量 SSM/SWA 做高效本地计算（几乎零通信）。这就是为什么 1M context 的 Nemotron Ultra、Gemma 3、Jamba 都选择了混合架构。

4. Megatron-Core 实现 — Gated DeltaNet 的 All-to-All CP

# ═══ MCore GDN CP 策略: All-to-All CP↔HP ═══
# 文件: megatron/core/ssm/gated_delta_net.py
# 核心思想: 不传 state，换维度！
# CP sharding (序列切分)
#   → All-to-All → HP sharding (head 切分)
# 每 rank 拿完整序列但只算部分 head
# 递推在完整序列上执行 → 无串行依赖！

# ═══ 非 CP 时的数据流 ═══
# 每 GPU 持有完整序列，全部 head (TP 内)
# shape: [seq_len, batch, hidden]
def forward_no_cp(hidden_states):
    # 1. 投影: h → q,k,v,gate,beta,alpha
    qkvzba, _ = in_proj(hidden_states)
    # shape: [S, B, qk*2+v*2+heads*2]

    # 2. Conv1d on QKV
    qkv = causal_conv1d(qkv, conv1d.weight)

    # 3. GDN 递推 (chunk_gated_delta_rule)
    # S_t = g_t⊙S_{t-1} + β_t·(k_t⊗v_t - (S·k_t)⊗k_t)
    out = chunk_gated_delta_rule(q, k, v, g, beta)

    # 4. Output norm + gate + proj
    out = gated_norm(out, gate)
    out = out_proj(out)
    return out

# ═══ CP 时的数据流 (实际代码) ═══
# 每 GPU 持有 S/P 的序列, 全部 head (TP 内)
def forward_with_cp(hidden_states):
    # shape: [S/P, B, hidden]

    # 1. 投影 (在 S/P 上做, 无通信)
    qkvzba, _ = in_proj(hidden_states)
    # shape: [S/P, B, qk*2+v*2+heads*2]

    # ★ 2. All-to-All: CP → HP
    # 输入: [S/P, B, H_all]  (序列分片)
    # 输出: [S, B, H_all/P]  (头分片)
    qkvzba = tensor_a2a_cp2hp(
        qkvzba,
        seq_dim=0, head_dim=-1,
        cp_group=pg_collection.cp,
        split_sections=[qk, qk, v, v, h, h],
    )
    # ↑ 每个 section 独立做 a2a
    # qk 维度: S/P → S, heads/P

    # 3. Conv1d (参数也要按 head 切片!)
    conv1d_weight = get_parameter_local_cp(
        self.conv1d.weight, dim=0,
        cp_group=pg_collection.cp,
        split_sections=[qk, qk, v],
    )
    # ↑ 只取当前 rank 负责的 head 对应参数
    qkv = causal_conv1d(qkv, conv1d_weight)

    # 4. A_log, dt_bias 也按 head 切片
    A_log_local = get_parameter_local_cp(
        self.A_log, dim=0,
        cp_group=pg_collection.cp,
    )
    # 只取 num_heads/cp_size 个 head 的参数

    # 5. GDN 递推 (完整序列, 部分 head)
    # 每 rank 独立递推 → 无串行依赖!
    out = chunk_gated_delta_rule(q, k, v, g, beta)

    # 6. Output norm + gate
    out = gated_norm(out, gate)

    # ★ 7. All-to-All: HP → CP
    # 输入: [S, B, V/P]  (头分片)
    # 输出: [S/P, B, V]  (序列分片)
    out = tensor_a2a_hp2cp(
        out, seq_dim=0, head_dim=-1,
        cp_group=pg_collection.cp,
    )

    # 8. Output proj (在 S/P 上做)
    out = out_proj(out)
    return out

MCore GDN: All-to-All 把序列分片换成 Head 分片，消除串行依赖

核心洞察: 不传 State，换维度

问题回顾: Section 2 分析了 Linear Attention CP 的串行依赖难题 — chunk 间有状态依赖，需要 parallel scan 或串行传 state。

MCore 的解法: 完全绕过这个问题！不在序列维度分片做递推，而是用 All-to-All 换成 Head 维度分片。每个 rank 拿到完整序列但只负责 H/P 个 head → 递推在完整序列上独立执行，零串行依赖。

非 CP vs CP 的 5 处关键区别

① 输入 shape: 非CP 拿完整 [S,B,H]；CP 时只拿 [S/P,B,H]。

② All-to-All 通信: CP 时在 in_proj 后插入 tensor_a2a_cp2hp，把 [S/P, B, H_all] 变成 [S, B, H_all/P]。GDN 核心计算完后再用 tensor_a2a_hp2cp 变回来。

③ 参数切片: conv1d.weight、A_log、dt_bias 都需要用 get_parameter_local_cp 按 head 维度取当前 rank 的切片。非 CP 时用完整参数。

④ 递推维度: 非 CP 在全 head 上递推；CP 时只递推 H/P 个 head，但是在完整序列上 → 没有 chunk 间的 state 传递。

⑤ 通信代价: 两次 All-to-All（前后各一次）。通信量 = 2 × S × H_all/P × sizeof(bf16)。这比 Ring Attention 的 P 步 KV 传递要大，但换来了零串行依赖。

为什么可行？

前提: GDN 的递推 \(S_t = g_t \odot S_{t-1} + \beta_t(k_t \otimes v_t)\) 中，不同 head 之间完全独立。Head 维度的并行是"天然免费的"。

约束: num_heads % (TP × CP) == 0。头数必须能被 TP×CP 整除。

与 Section 2 的修正

之前说: "DeltaNet 不满足结合律 → parallel scan 不适用 → 只能串行"。

修正: MCore 用的不是 parallel scan，而是All-to-All 换维度。代价是通信量更大（整个 activation tensor 做 a2a），但完全消除串行等待。这是在"通信量"和"并行度"之间做的 tradeoff — 对 NVLink 高带宽场景非常友好。

设计哲学: Ring Attention 是"Q 不动，KV 来找 Q"（序列并行+通信KV）。GDN 的 All-to-All CP 是"序列不分，Head 来分"（维度交换+独立递推）。两种完全不同的并行化思路，各自适合不同的 operator pattern。
