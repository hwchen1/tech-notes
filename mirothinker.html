1. 核心论点 — Scaling "有效交互" 而非交互长度 + 双循环 ReAct

论文的反直觉论点:单纯把推理轨迹拉长(更多 step / tool call)不能可靠提升长程任务表现——当中间步骤不准、grounding 差时,更长的轨迹只会累积噪声、传播错误,最终拖垮答案质量。真正该 scale 的是有效交互 (effective interaction),它依赖两件事:(1) 每一步的原子 agentic 能力(planning / reasoning / tool 执行都要强);(2) 推理过程中能验证并修正轨迹的机制。前者由 MiroThinker-1.7 的 agentic mid-training 解决,后者由 H1 的 verification 模式解决。

工作流基于 ReAct 扩展成双循环:外层 episode loop 负责轨迹级重启(超 $T_{max}$ 或格式错误就丢弃全部上下文、用原始 query 干净重启,最多 $R_{max}$ 次);内层 step loop 驱动 thought→action→observation。关键工程点是上下文算子 $\Phi_t$:完整保留 thought/action 轨迹,但对 observation 只保留最近 $K{=}5$ 步(sliding window)+ 每条截断到 $L$ token,在固定预算内撑起超长轨迹。

# 双循环 ReAct + 上下文管理

for e in range(R_max):        # 外层 episode loop
    C = {query}                # 干净重启, 丢弃前轨迹
    for t in range(T_max):    # 内层 step loop
        # 上下文算子 Φ: thought/action 全留
        #   observation 只留最近 K=5 步 + 截断 L
        C = build_ctx(traj, K=5, L=trunc)

        T = f(query, C)        # thought
        A = pi(C, T)           # action (tool call)
        O = Tool(A)            # observation
        traj.append((T, A, O))
        if is_final(A): return A

    # 超 T_max 未出答案 → 进入下一 episode
# 最后一个 episode 强制出答案(取最佳中间答案兜底)

关键 insight

有效交互 > 交互长度: 长程任务的瓶颈不是 step 不够,而是每步质量。差的步骤越多越糟 → 先把单步做对,再谈 scaling。
Sliding window K=5: 砍掉远端 observation 几乎不掉点,却能撑起远更长的轨迹。关键是thought/action 全留,模型仍能回看早期决策,只是把"看证据"的窗口收紧到最近 5 步。
Episode 干净重启: 一条轨迹陷入死胡同/超预算时,与其在污染的上下文里硬撑,不如丢光重来。最后一轮强制出答案+最佳中间答案兜底,绝不静默失败。
三类工具: 检索 (google_search + scrape, 后接小模型蒸馏证据)、代码执行 (E2B sandbox)、文件传输。框架层自动纠正 malformed tool call,避免长轨迹被累积失败带偏。

2. 四阶段训练流水线 — Mid-train → SFT → DPO → GRPO

从开源 Qwen3-MoE 起步,四阶段层层递进:① Agentic Mid-training(强化原子能力)→ ② SFT(学结构化 agentic 交互)→ ③ DPO 偏好优化(对齐决策偏好)→ ④ GRPO 在线 RL(真实环境试错探索)。其中 mid-training 是关键创新:不监督整条轨迹,而是抽出单步 $k$ 重写成高质量目标(planning / reasoning / summarization),只对该步算 next-token loss。

四阶段损失:Mid-train 单步监督 $\mathcal{L}_{mid}=-\mathbb{E}[\log\pi_\theta(y_k\mid C_{\lt k})]$;SFT 整轨迹模仿 $\mathcal{L}_{SFT}=-\mathbb{E}\sum_t\log\pi_\theta(T_t,A_t\mid x,H_{\lt t})$(tool 输出当 user turn,不真跑工具);DPO 仅以答案正确性排序(不强加结构约束)$\mathcal{L}_{PO}=\mathbb{E}[\mathcal{L}_{DPO}]+\lambda\mathcal{L}_{SFT}^{(+)}$;GRPO 组内相对优势 $\hat{A}_i=R_i-\frac1G\sum_j R_j$,配 entropy control 防熵崩。

# 四阶段流水线 (从 Qwen3-MoE)

# ① Agentic Mid-training — 强化原子能力
#   抽单步 k 重写, 只对该步算 loss
y_k = rewrite(traj[k], ctx=C[:k])  # planning/reason/summ
L_mid = -log pi(y_k | C[:k])       # k=1: 冷启 planning
#   混入通用指令数据防遗忘

# ② SFT — 整轨迹模仿 (tool输出当user turn)
L_sft = -sum(log pi(T_t, A_t | x, H[:t]))

# ③ DPO — 只按答案对错排序, 不限结构
L_po  = L_dpo(x, H_pos, H_neg) + lam * L_sft_pos
#   mini 版用 preference distillation 蒸强模型偏好

# ④ GRPO — 在线 RL, 组内相对优势
A_hat = R(x,H) - mean(R over G rollouts)
#   R = a_c·correct - a_f·format
#   entropy control: 对负rollout低概率token加KL罚
#   streaming rollout + 长尾优先调度

训练关键点

Mid-train 抽单步重写: 不监督整条轨迹(噪声大),而是 isolate 第 $k$ 步、conditioned on 完整前缀重写成高质量目标,只对该步算 loss。冷启 planning ($k{=}1$)+ 中间 reasoning/summary ($k{>}1$) 一网打尽,且随机施加上下文压缩练鲁棒性。
DPO 不限结构: 只用答案正确性当排序信号,刻意不强加 planning 长度/step 数/模板等约束——这类硬约束会引入系统偏差、损泛化。mini 版额外用 preference distillation 从强模型蒸偏好。
GRPO entropy control: 对负 rollout 里的低概率 token 加辅助 KL 罚,防止模型持续压低这些 token 概率导致过早熵崩,维持探索。
双 pipeline QA: Corpus-based(Wikipedia/OpenAlex 子图采样,高吞吐广覆盖)+ WebHop(answer 为根的推理树 + web 扩展 + 分层可解性验证 + 叶子混淆防泄答案)。难度自适应过滤:弱 agent 能解的进 SFT,强 agent 解不动的留给 RL。

3. Heavy-Duty Reasoning Mode (H1) — Local + Global 双层验证

H1 在 MiroThinker-1.7 之上把 verification 直接嵌入推理过程,是论文首次系统性探索"验证驱动的长程推理"。两个独立模块:Local Verifier 审计步级决策,Global Verifier 审计整条证据链。

Local 验证:标准 ReAct 天然走模型给出的最高概率路径,在难题上这种"概率偏好"会把 agent 带进习惯性思维定式——不停确认自己的偏好而非真正探索。Local Verifier 在中间步(planning 决策、tool 调用、假设更新)介入,促使 agent 更彻底地探索解空间、主动从环境取反馈、早期就纠错。Global 验证:利用"验证比生成容易"这一长期被忽视的非对称性,在出答案前组织全部已收集证据;若证据不足,要求 agent 重采样/补全推理链而非草率给答案;在可控算力预算内,最终选出证据最完整可靠的那条答案路径。

# H1: verification-centric reasoning

# ===== Local Verifier (步级) =====
for t in step_loop:
    T = think(C)
    A = action(C, T)
    # 标准 ReAct: 直接执行最高概率 A → 易定式
    feedback = local_verify(T, A, env)
    if feedback.should_explore:
        A = reconsider(C, alternatives)  # 探索替代动作
    O = Tool(A)

# ===== Global Verifier (轨迹级) =====
init_ans = summarize(trajectory)
evidence = organize_evidence(trajectory)
if not sufficient(evidence):
    # 证据不足 → 不草率出答案
    resample_or_extend(agent)            # 补全推理链
else:
    # 多候选路径里选证据最完整可靠的
    final = select_best_grounded(paths)
# 利用 "验证 < 生成" 的非对称性

验证模式关键点

对抗概率偏好: ReAct 沿最高概率走,在难题上退化成"反复确认自己的偏好"。Local Verifier 强制 agent 跳出定式、主动探索替代动作并向环境要反馈。
验证 < 生成 的非对称性: 判断一条证据链是否支撑答案,比从头生成答案容易得多。Global Verifier 把这个便宜的"验证"用在出答案前——证据不够就回去补,够了就在多候选里挑最 grounded 的。
不草率终止: 与一般 agent "凑够轮数就交差"相反,H1 显式检查证据充分性,宁可重采样/延长也不交一个证据不足的答案 → 这正是 "heavy-duty"(重载)的含义。
能力 × 验证 解耦: 1.7 负责"每步做对"(原子能力),H1 的两层验证负责"整体不跑偏"(可靠性)。两者正交叠加,在可控算力预算内换取长程推理的可靠性。
