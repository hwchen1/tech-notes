MQT — Matryoshka Query Transformer

可变长 Query 压缩视觉 Token | Matryoshka 式嵌套训练 | 2→256 tokens 按需截断 | LLaVA-OneVision

1. 核心理念：可变长视觉 Token 压缩

# MQT 核心思路:
# ViT 输出 576 visual tokens 太多
# 用 M 个 learnable queries 压缩
# 训练时随机截断 → 推理时按需选择

# 标准 VLM 流程:
# ViT: image → 576 tokens
# MLP: 576 tokens → LLM (太长!)

# MQT 流程:
# ViT: image → 576 tokens
# MQT: 576 → M tokens (M=2~256)
# LLM: 只处理 M 个压缩 token

visual_tokens = vit(image)  # [576, d]
queries = learnable(256, d)  # [256, d]

# Cross-attention 压缩
compressed = cross_attn(
    Q=queries,        # [256, d]
    K=visual_tokens,  # [576, d]
    V=visual_tokens   # [576, d]
)  # → [256, d]

# 推理时截断前 m 个 (m ≤ 256)
output = compressed[:m]  # [m, d]

MQT 架构：ViT → 576 tokens → MQT 压缩 → m 个 token 送入 LLM

为什么需要 MQT？

标准 VLM 中 ViT 输出 576 个 visual tokens，占据 LLM 输入的大量位置。对于简单图像（如单个物体），576 tokens 是浪费；对于复杂文档，可能还不够。

MQT 实现"按需分配"：简单图像用 16 tokens，复杂图像用 256 tokens，计算量与图像复杂度成正比。

与 Q-Former 的区别：Q-Former 固定 32 个 query，不可变。MQT 训练时用 Matryoshka 策略让前 m 个 query 也能独立工作。

2. Matryoshka 训练策略

# Matryoshka Representation Learning
# 灵感来源: 俄罗斯套娃
# 外层包含内层的所有信息

# 训练时: 随机采样 m ∈ {2,4,8,...,256}
# 只用前 m 个 query 的输出训练
# → 前 m 个 query 学会独立工作!

class MQT(nn.Module):
    def __init__(self, M=256, d=1024):
        self.queries = nn.Parameter(
            torch.randn(M, d))
        self.cross_attn = CrossAttention(d)
        self.mlp = MLP(d, d_llm)

    def forward(self, visual_tokens, m=None):
        # visual_tokens: [B, 576, d]
        compressed = self.cross_attn(
            Q=self.queries,  # [256, d]
            KV=visual_tokens # [576, d]
        )  # → [B, 256, d]

        # 训练: 随机截断
        if self.training:
            m = random.choice(
                [2,4,8,16,32,64,128,256])
        # 推理: 用户指定 m
        else:
            m = m or 256

        return self.mlp(compressed[:, :m])

Matryoshka 套娃结构：前 m 个 query 自包含

为什么这样训练有效？

随机截断迫使模型将最重要的信息编码到前几个 query 中。前 2 个 query 捕捉类别和主体；前 16 个捕捉空间关系；前 256 个保留完整细节。

$\mathcal{L}_{\text{MQT}} = \mathbb{E}_{m \sim \mathcal{M}} \left[ \mathcal{L}_{\text{LLM}}(y \mid x_{\text{text}}, z_{1:m}) \right]$

其中 \(\mathcal{M} = \{2, 4, 8, 16, 32, 64, 128, 256\}\)，\(z_{1:m}\) 是前 m 个压缩 token。

信息密度递减：Query 1-2 携带 ~70% 的分类信息，Query 3-16 补充空间细节，Query 17-256 增加纹理/边缘等冗余信息。

3. 架构细节与设计选择

# MQT 的 Cross-Attention 结构
# 2层 transformer + cross-attn

class MQTLayer(nn.Module):
    def __init__(self, d, n_heads):
        # Self-attention among queries
        self.self_attn = MultiHeadAttn(d, n_heads)
        # Cross-attention: queries → visual
        self.cross_attn = MultiHeadAttn(d, n_heads)
        self.ffn = FFN(d, d * 4)

    def forward(self, queries, visual):
        # Step 1: queries 互相交流
        q = self.self_attn(queries, queries)
        # Step 2: queries 从 visual 提取信息
        q = self.cross_attn(q, visual)
        # Step 3: FFN 非线性变换
        return self.ffn(q)

# 完整 MQT: 2层堆叠
# 参数量极小: ~50M
# (对比 ViT-L: 307M)

# 位置编码: 2D sinusoidal
# queries 本身有 learned pos
# visual tokens 继承 ViT 的 pos

MQT Layer: Self-Attn (query间) + Cross-Attn (query←visual) + FFN

为什么只需 2 层？MQT 不需要"理解"图像（ViT 已经做了），只需要"压缩"——将 576 维的冗余信息聚合到 M 个 query 中。压缩是比理解简单得多的任务。

4. 与 Omni-Encoder 的对比

# ========== MQT 方案 ==========
# 保留 ViT，加一个轻量压缩层
image → ViT(307M) → 576 tokens
     → MQT(50M) → m tokens
     → LLM(7B) → answer

# 总视觉参数: 307M + 50M = 357M
# 压缩比: 576/m (最高 288x)
# 优点: 轻量, 即插即用
# 缺点: 仍需 ViT 前置推理


# ========== Omni-Encoder 方案 ==========
# 完全替代 ViT，LLM 自身压缩
image → PatchEmbed → raw tokens
     → MoE-LLM(3B/500M)
     → queries 位置输出 = 压缩

# 总视觉参数: 3B (激活 500M)
# 压缩比: 2704/256 = 10x (Q1)
#    or 2704/8192 → 进一步 LLM
# 优点: 无 ViT, 一体化
# 缺点: 训练成本高 (1000×H200)


# ========== Q-Former 方案 ==========
# 固定 32 queries, 不可变
image → ViT → 257 tokens
     → Q-Former(32 queries)
     → 32 tokens → LLM

# 总: ViT + Q-Former (~130M)
# 压缩比: 固定 257/32 = 8x

MQT vs Omni-Encoder vs Q-Former 三种视觉 Token 压缩范式对比

哲学差异

MQT 的哲学："ViT 已经很好了，问题只是 token 太多。加一个轻量层压缩一下就行。"

Omni-Encoder 的哲学："为什么要两个模型？一个 LLM 能同时做编码和压缩，还能处理所有模态。"

MQT 的优势：极低成本实现可变压缩。只需训练 50M 参数的 MQT 模块，就能把任意 VLM 的 visual tokens 从 576 压缩到 2-256 个。
Omni-Encoder 的优势：架构更统一、更简洁。不需要维护 ViT + MQT + LLM 三个组件，一个 MoE-LLM 搞定一切。

5. 性能：Token 数 vs 精度权衡

# MQT 在不同 token 数下的表现
# (LLaVA-1.5-7B + MQT)
#
# Tokens  VQAv2   GQA    TextVQA
# ------  -----   ---    -------
# 576     78.5    62.0   58.2   (baseline)
# 256     78.3    61.8   57.9   (-0.2%)
# 144     77.9    61.5   57.1   (-0.6%)
# 64      77.1    60.8   55.6   (-1.5%)
# 16      74.8    58.9   51.2   (-4.7%)
# 4       70.2    55.1   43.8   (-10.5%)
# 2       65.1    51.3   38.4   (-17.2%)

# 关键发现:
# 64 tokens (压缩 9x) 只掉 1.5%
# 16 tokens (压缩 36x) 只掉 4.7%
# Sweet spot: 64~144 tokens

# 推理加速:
# 576 tokens: prefill 12.3ms
# 64 tokens:  prefill 4.1ms (3x↑)
# 16 tokens:  prefill 2.8ms (4.4x↑)

# 因为 LLM self-attn 是 O(N²)
# token 减少 9x → 计算减少 ~81x
# (但还有 text tokens, 所以实际 3x)

Token 数 vs 精度：16~144 tokens 是 sweet spot (精度损失 <5%)

核心发现

256 tokens 几乎无损：从 576 压缩到 256 (2.25x)，VQAv2 只掉 0.2%。大量视觉 token 是冗余的。
64 tokens 是最佳性价比：9x 压缩只掉 1.5% 精度，但推理速度提升 3x。对于大部分实际应用足够。
极端压缩（2 tokens）仍可用：只用 2 个 token 时 VQAv2 仍有 65%——对分类、物体识别等粗粒度任务足够。

应用场景匹配：

• 分类/检测 → 2-16 tokens

• VQA/对话 → 64-144 tokens

• OCR/文档 → 256 tokens
