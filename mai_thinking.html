核心设计理念

Microsoft AI 的架构选择不追求新颖性，而是追求可验证的 scaling 效率：

Scaling Ladder 方法论：每个决策（MoE 稀疏度、注意力模式、FFN 配置）都通过 8 个量级的 ladder 模型（365M → 35B）验证 scaling 性质。只有在所有尺度都持续有效的改进才被采纳。

Efficiency Gain (EG) 指标：

\( EG = f^{-1}(L') / C' \)

基线需要多少额外计算才能达到候选模型的损失？EG=1.3 意味着基线需要 30% 更多计算。

NoPE 的采用：Global attention 层使用无位置编码 (Kazemnejad et al., 2023)，在 scaling ladder 上性能与 RoPE 相当但更高效。5:1 local:global 比例大幅减少了 KV cache 开销。
LatentMoE：在 all-to-all dispatch 之前施加 shared down-projection 将 token 压缩到低维空间（3072 → experts），减少通信量。与 NVIDIA 联合设计。

2. 核心方法：Hill-Climbing Machine

# Hill-Climbing = 持续 log-linear RL 改进
# 核心哲学: 进步不靠单个模型突破
#   而是构建一个能持续改进的系统

# ====== RL 目标 (GRPO-based) ======
# J(θ) = E[ Σ min(r·A, clip(r)·A) ]
# r_it = π_θ(y_it|q,y_<t) / π_old(...)
# A_i = (R_i - mean(R)) / std(R)
# Token-level policy gradient
# 每个 token 贡献相等 (不管长度)

# ====== Reward 分解 ======
# R(q,y) = R_task + w_lang·R_lang
#          - w_len·R_len
# R_task: 领域相关 (测试/judge)
# R_lang: 语言一致性 (防 CoT 混语言)
# R_len: 长度惩罚 (ρ·|y|/ℓ_max)

# ====== 稳定性创新 ======

# 1. Adaptive Entropy Control
# k ← clip(k + δ·sign(H* - Ĥ), 0, k_max)
# 动态调整 clip 上界 → 自动熵正则
# 熵太低→放宽上界→鼓励探索
# 熵太高→收紧上界→稳定训练

# 2. Outer Ratio Clip
# 对所有分支施加硬 clip [r_min, r_max]
# 防止极端 importance ratio → 梯度爆炸

# 3. Self-Distillation Recovery
# 爬坡崩溃时 → 收集最优轨迹 (~1M)
# → SFT 回到稳定点 → 继续 RL 爬坡
# 也用于跨代基模型迁移

# 4. Problem Sampling Filter
# pass_rate ∈ [ρ_min, ρ_max] 才训练
# 太简单/太难的 = 低方差 = 无用

三个领域专家爬坡 → SFT 合并 → 最终爬坡。Self-distillation 实现崩溃恢复和跨代迁移

Hill-Climbing 核心理念

MAI-Thinking-1 的核心贡献不是单一的模型创新，而是构建了一个能持续改进的系统：

三大设计原则：

1. Capabilities should be learned, not inherited — 不蒸馏第三方模型

2. Simplicity is sustainable — 简单可扩展的 recipe

3. Scientific rigor avoids shortcuts — 所有决策通过 ladder 验证

Self-Distillation 的作用：(1) 爬坡崩溃时恢复进度 (2) 新基模型上迁移旧爬坡成果 (3) 过滤 reward hacking 样本。~1M traces 足以匹配教师性能。

Adaptive Entropy Control

RL 爬坡的最大挑战是熵崩溃（模型过早收敛到单一策略）。MAI 用积分控制器动态调整 clip 上界：

\( k \leftarrow \text{clip}(k + \delta \cdot \text{sign}(H^* - \hat{H}(\pi_\theta)),\, 0,\, k_{\max}) \)

这比显式 entropy bonus 更稳定，避免了超参敏感性。

为什么不需要 Reward Model：STEM 域用代码执行验证正确性；Agentic 域用测试通过率作 reward。只有 Helpfulness 域需要 AI judge。Ground-truth reward 无法被 hack。

3. Agentic Climb：多步环境交互 RL

# Agentic RL 与标准 STEM RL 的区别:
# STEM: 单轮生成 → 验证答案
# Agentic: 多步交互 → 环境反馈 → 迭代

# ====== Multi-step 框架 ======
# Episode = trajectory of steps:
#   Step 1: model → tool_call → observe
#   Step 2: model → tool_call → observe
#   ...    (直到完成或超时)
#   Final: reward = test_pass_rate

# 工具集 (SWE environments):
# - bash (终端命令)
# - string_replace (文件编辑)
# 只用这两个工具, 极简但强大

# ====== SWE 环境构建 ======
# 1. 从 GitHub repos 构建容器
# 2. 每个环境 = 完整 repo + 依赖 + 测试
# 3. 自动 setup/teardown
# 4. 测试套件 = 自动 reward 信号

# ====== 混合训练 ======
# Agentic climb 混合 STEM 任务:
# - STEM 帮助稳定训练
# - STEM → Agentic 正迁移
# - Agentic → STEM 无负迁移
# 最终: agentic + coding 联合爬坡

# ====== Reward 设计 ======
# SWE: 测试通过率 (verifiable)
# Tool use: AI judge (行为质量)
# 混合: verifiable + judge reward
# 关键: 测试结果 = ground truth
#   不可被 gaming/hacking

Agentic climb 在容器化环境中执行多步交互，reward 来自测试通过率 + AI 评判

SWE 环境基础设施

Agentic RL 的瓶颈是环境构建。Microsoft 投入大量基础设施：

组件说明
SEESandbox Execution Environment
容器化每个 repo = 独立 Docker 环境
测试套件自动提取 + 验证可用
并行度大规模并发环境
工具bash + string_replace (极简)
关键发现：STEM 混合训练。单独做 Agentic climb 不稳定。混合 STEM 任务后训练更稳定，且 STEM → Agentic 有正迁移，反之无负迁移。

涌现行为

随着 RL 爬坡推进，模型行为持续演化（论文 Appendix C.2）：

• 早期：逐行阅读，盲目修改

• 中期：学会 grep 定位，增量验证

• 后期：全局分析，最小修改，错误恢复

这些策略不是 SFT 教的——完全从 RL 中涌现。

极简工具哲学：只给 bash + string_replace 两个工具。模型自己学会用它们组合出 grep、cat、git diff、pytest 等复杂操作序列。工具越简单 → 组合空间越大 → RL 越能探索创新策略。
