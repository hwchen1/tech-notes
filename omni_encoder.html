范式转变

传统: ViT + adapter + LLM (各模块分离)

Omni-Encoder: [raw + queries] → MoE-LLM → 取 query 位置输出

核心洞察: Causal Query

把 learnable queries 放在 raw tokens 后面，causal attention 让 queries 能看到所有 raw tokens — 这等效于 cross-attention，但无需额外模块！

$\text{input} = [\underbrace{r_1, r_2, \ldots, r_N}_{\text{raw tokens}}, \underbrace{q_1, q_2, \ldots, q_M}_{\text{learnable queries}}]$ $\text{output} = \text{MoE-LLM}(\text{input})[-M:]$ $M = N / 10 \implies 10\times \text{ compression}$

为什么这比独立 cross-attn 更优？

1. 无额外模块 — 压缩完全在 LLM 内完成

2. 与 decoder 完全同构 — 同 kernel, 同 TP/PP 策略

3. LLM 世界知识参与压缩 — 不是盲目池化，是"理解后压缩"

4. 更好优化 — PPT: "与 Decoder（LLM 一致）框架更好优化"

2. Causal Query：为什么 Causal Attention 能做压缩

# Causal Query 原理详解
# 核心: 位置决定了"谁能看到谁"

# 输入序列 (拼接后):
# pos:  0   1   2  ... 2703  2704 2705 ... 2959
# tok: [r0, r1, r2, ..., rN, | q0,  q1, ..., qM]
#       ← raw tokens →        ← queries →

# Causal Mask (下三角):
# r0 能看到: r0
# r1 能看到: r0, r1
# r2 能看到: r0, r1, r2
# ...
# q0 能看到: r0, r1, ..., rN, q0  ← 全部!
# q1 能看到: r0, r1, ..., rN, q0, q1
# ...

# 关键: query 在后面 →
# 它能 attend to 所有 raw tokens
# 这和 cross-attention 效果一样!

# 但有额外好处:
# 1. raw tokens 之间也互相交互
#    (自回归式 feature extraction)
# 2. queries 之间也互相交互
#    (信息在 queries 间流动)
# 3. 全部在一次 forward 完成

# 实现极简:
class OmniEncoder(nn.Module):
    def __init__(self):
        self.llm = MoELLM("3B-A500M")
        self.queries = nn.Parameter(
            torch.randn(max_queries, d_model)
        )

    def forward(self, raw_tokens):
        n_q = len(raw_tokens) // 10
        q = self.queries[:n_q]

        # 拼接: [raw | queries]
        seq = torch.cat([raw_tokens, q])

        # 一次 LLM forward (causal)
        out = self.llm(seq)

        # 只取 query 位置
        return out[len(raw_tokens):]

Causal Query = 位置魔法

不需要设计独立的 cross-attention 模块。只需把 learnable queries 拼接在 raw tokens 后面，causal mask 自动保证 queries 能看到所有 raw tokens。

$\text{Causal Mask: } \text{Attn}(i,j) = \begin{cases} \text{allow} & j \leq i \\ \text{block} & j > i \end{cases}$ $q_k \text{ at pos } (N+k) \text{ can attend to all } r_0 \ldots r_N$

比 Cross-Attention 更强的地方

1. Raw tokens 互相交互: r1 能看到 r0，r2 能看到 r0/r1... 相当于自回归 feature extraction。

2. Queries 互相交互: q1 能看到 q0，避免信息重复。

3. LLM 世界知识参与: 压缩不是盲目 pooling，而是带推理的信息提取。

动态 Query 数量

压缩比恒定 10x，query 数 = N/10。不同分辨率自动适应：

728² → 2704/10 = 256 queries

1456×728 → 5408/10 = 540 queries

364² → 676/10 = 68 queries

PPT 算法优点

1. 文本模态引入，增加多模态 encoder 必要性

2. Query 数量调节，二维 token 不再被 fix

3. 一套 Query token 作为万物 representation

4. 与 Decoder（LLM 一致）框架更好优化

3. 双 Query 设计与压缩比详解

# 双 Query: 语义 + 空间 两条路径
# 都通过 causal query 机制实现

# Query 1: 高压缩, 语义路径
input1 = cat(raw, query1)  # [2704+256]
out1 = llm(input1)[-256:]  # 10x 压缩

# Query 2: 低压缩, 空间细节
input2 = cat(raw, query2)  # [2704+8192]
out2 = llm(input2)[-8192:]  # ~2x 压缩

# ============================
# PPT 标注是【端到端总压缩比】
# 不是 Query 自身的压缩比!
# ============================

# Method 1: LLM 做 history 压缩
# Q1: encoder 10x → history 8x → 总 80x
# Q2: encoder ~2x → 不做 history → 总 2x
# 用途: 多图/长视频/多轮对话

# Method 2: LLM 不压缩
# Q1: encoder 10x → 不压缩 → 总 10x
# Q2: encoder x4 扩展 → 不压缩 → 总 x4
# 用途: 高分辨率/OCR/空间推理

# 为什么同样的 Q1=256:
# Method 1 标 "80x", Method 2 标 "10x"?
# 因为标注是端到端压缩:
#   M1: 10x(enc) × 8x(history) = 80x
#   M2: 10x(enc) × 1(不压) = 10x

# History 8x 压缩:
# 多轮对话中, 旧 turn 的 256 tokens
# 被 "text compression latent modal"
# 进一步压缩 → ~32 tokens/图

双 Query 优势

Q1 (语义): 10x 压缩，保留全局语义，跨 turn 持久记忆。

Q2 (空间): ~2x/x4，保留空间细节，当前 turn 用后丢弃。

$\text{M1 Q1: } 2704 \xrightarrow{10\times} 256 \xrightarrow{8\times} 32 = 80\times$ $\text{M2 Q1: } 2704 \xrightarrow{10\times} 256 \xrightarrow{1\times} 256 = 10\times$

"近处精细, 远处模糊"

当前图: Q1+Q2 全量 (8448 tokens)

历史图: 只保留 Q1 压缩后 (32 tokens)

4. 训练流程 (端到端 NTP)

# 训练 pipeline (单样本)
# Encoder = Decoder = 同一个 MoE-LLM

class OmniEncoderTrainer:
    def __init__(self):
        self.tokenizer = PatchEmbedding(p=14)
        self.llm = MoELLM("3B-A500M")
        # 可学习 queries (唯一新增参数)
        self.q1 = Parameter([max_q1, d])
        self.q2 = Parameter([max_q2, d])

    def train_step(self, image, text):
        # 1. Tokenize
        raw = self.tokenizer(image)  # [2704, d]

        # 2. Encoder: [raw | queries] → LLM
        n_q1 = len(raw) // 10  # 256
        enc_input = cat(raw, self.q1[:n_q1])
        enc_out = self.llm(enc_input)
        compressed = enc_out[len(raw):]
        # [256, d] — 只取 query 位置

        # 3. Decoder: [compressed | text] → LLM
        dec_input = cat(compressed, text)
        logits = self.llm(dec_input)

        # 4. NTP Loss (text 部分)
        text_logits = logits[256:]
        loss = F.cross_entropy(
            text_logits[:-1], text[1:]
        )

        # Backward: 梯度通过两条路径
        loss.backward()
        # Path A: loss→llm(dec)→∂L/∂W
        # Path B: loss→llm(dec)→compressed
        #         →llm(enc)→∂L/∂W (叠加!)
        # Path C: →queries→∂L/∂W_q
        # Path D: →raw→tokenizer→∂L/∂W_tok

        return loss

5. Inference 与 KV Cache 管理

# Inference 时 Q1 + Q2 永远都执行
# 区别在 KV cache 生命周期

class OmniInference:
    def encode(self, image):
        raw = patch_embed(image)  # [2704, d]
        n_q1 = len(raw) // 10

        # Q1: [raw | q1] → LLM → 取 query 位
        seq1 = cat(raw, self.q1[:n_q1])
        enc_q1 = self.llm(seq1)[len(raw):]
        # [256, d]

        # Q2: [raw | q2] → LLM → 取 query 位
        seq2 = cat(raw, self.q2[:n_q2])
        enc_q2 = self.llm(seq2)[len(raw):]
        # [8192, d]

        return enc_q1, enc_q2

    def build_llm_input(self, history, cur):
        # 历史: Q2 丢弃, Q1 压缩 8x
        hist = []
        for turn in history:
            h = history_compress(turn.q1, 8)
            hist.append(h)  # 256→32

        # 当前: Q1+Q2 全量
        cur_tok = cat(cur.q1, cur.q2)
        # = 8448 tokens

        # 拼接
        llm_in = cat(*hist, cur_tok, prompt)
        return llm_in

    def generate(self, llm_in):
        return self.llm.generate(llm_in)

# 10 轮对话 token 分析:
# 不压缩: 10×8448 = 84,480 → OOM
# Omni:   9×32 + 8448 = 8,736 → OK

Q1/Q2 永远都执行

不存在"按需启用 Q2"，否则 train/test 分布偏移。

$\text{当前: } 256 + 8192 = 8448$ $\text{历史: } 256 / 8 = 32$ $\text{10 轮: } 8448 + 9 \times 32 = 8736$

为什么不保留 Q2 历史？

10 图 × 8192 = 81,920 tokens → OOM。旧图空间细节价值低。

计算开销

Encoder 处理 [raw(2704) + queries(256)] = 2960 tokens 的 causal attention。比传统的 2704 tokens 略多，但在同一次 forward 完成压缩。

# 架构范式对比

# LLaVA: ViT + MLP → LLM
feat = vit(image)         # [576, d]

# BLIP-2: ViT + Q-Former → LLM

    queries,              # [32, d]
    kv=feat               # [576, d]

# Flamingo: ViT + Perceiver → LLM
feat = vit(image)

    latents,              # [64, d]
    kv=feat

# Omni-Encoder: 无独立模块!
raw = tokenize(image)     # [2704, d]

out = moe_llm(cat(enc, txt))

范式差异

LLaVA: 不压缩，所有 ViT token 直传 LLM。简单但多轮会 OOM。

BLIP-2 / Flamingo: 独立 cross-attention 模块做压缩。额外参数 + 与 LLM 异构。

Omni-Encoder: 零额外模块，causal attention 在 LLM 内部完成压缩。

Omni 独特优势

1. 完全同构 — encoder = decoder = 同架构

2. 全模态原生 — 不限于视觉

3. 带推理的压缩 — LLM 世界知识参与

4. 压缩比可调 — 10x~80x 按需

Omni 代价

训练成本量级差距：1000×H200 vs 8×A100

LLM causal mask 限制全局视觉交互

无 ViT 预训练基础特征可复用

8. 范式之争：ViT+LM vs LM+LM

# ========== 方案 A: ViT + LM ==========
# 当前主流 (LLaVA, Qwen-VL, GPT-4V)
# 专用 encoder + 通用 decoder

image → ViT(300M, bidirectional)
      → 576 visual tokens
      → MLP / MQT (可选压缩)
      → LLM(7B~72B) → text answer

# 优势:
# 1. ViT 有对比学习先验 (CLIP/SigLIP)
# 2. Bidirectional attn → 全局交互
# 3. 直接复用预训练权重 → 数天搞定
# 4. 编码阶段计算高效 (300M vs 3B)


# ========== 方案 B: LM + LM ==========
# Omni-Encoder 路线
# 统一 LM 做 encoder + decoder

image → PatchEmbed(几M)
      → raw tokens + learnable queries
      → MoE-LLM(3B, act 500M, causal)
      → compressed tokens
      → LLM(decoder) → text answer

# 优势:
# 1. 架构统一 → 工程简洁
# 2. 天然可变压缩 (causal query)
# 3. 多模态零成本扩展
# 4. 规模效应: LM 越大编码越强

两种范式的全面对比：ViT+LM 短期最优，LM+LM 长期可能胜出

ViT+LM 核心优势

信息密度高：Bidirectional attention 让每个 token 看到所有其他 token → 576 tokens 包含完整全局信息。

站在巨人肩膀：CLIP/SigLIP/DINOv2 大规模预训练免费可用，新 VLM 只需训 adapter。

编码效率：ViT-L 300M 参数对比 LM encoder 3B+，同样 576 tokens 计算量差 10x。

LM+LM 核心优势

架构统一：不需要维护 ViT 的特殊 pos encoding、不同 norm/activation、独立训练 recipe。

天然压缩：Causal mask + queries 后置 = 零额外模块实现可变压缩。

多模态零成本：新模态只需新 tokenizer，同一 LM 处理所有 token 序列。

无 Scale 天花板：LM encoder 规模可以无限增长，而 ViT 在 ~6B 后收益递减。

时间线判断

短期 (2024-2025)：ViT+LM 仍是最优解。SigLIP/DINOv2 太好，100x pretrained ViT 免费可用。
长期 (2026+)：LM+LM 可能胜出。算力不再是瓶颈时，统一性和多模态扩展性优势越来越大。类似 Transformer 统一 NLP 和 CV。
