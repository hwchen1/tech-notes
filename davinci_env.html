1. 问题与定位

训练强 SWE 智能体需要大规模、可执行、可验证的环境（Docker 里能编译、跑测试、观察运行时反馈的闭环）。但现状是两难：开源数据集规模与仓库多样性都有限，工业方案虽有规模却基础设施不开放。OpenSWE（论文即 daVinci-Env）要同时解决规模与质量/难度两个瓶颈，并把整套合成流水线全部开源。

两个瓶颈

规模瓶颈：构建验证过的环境计算/基础设施成本极高，学术组基本被挡在门外；工业方案闭源。

质量瓶颈：从真实仓库合成的环境常有 PR-Issue 错配（补丁并没解决 issue）或 过于平凡（issue 直接给出答案），要么不可解要么无学习信号。

贡献：① 全透明的空前规模（45,320 环境 + 完整流水线开源）；② 难度感知筛选管线；③ 强实证——SFT SOTA、log-linear 无饱和、且跨域正迁移。基座用 Qwen2.5-32B/72B-Base，非 Coder 版仍夺 SOTA。

2. 多智能体合成流水线

核心是把"从 GitHub PR 造出可验证 Docker 环境"这件苦活拆给一组协作智能体：先四段过滤挑出合格 PR，再由仓库探索 → Dockerfile → 评测脚本 → 测试分析四个智能体在一个迭代反馈闭环里协同产出。构建模型用 DeepSeek-V3.2。

# ① 四段 PR 过滤
keep = pr for pr in all_prs if (
    repo.stars >= 5          # 仓库成熟度
    and repo.lang == "Python"   # 语言
    and pr.has_issue_desc      # 有问题陈述
    and pr.nontest_code_diff)  # 改的是实现非测试

# ② 四智能体迭代闭环
ctx = explore_agent(repo)      # README/CI/依赖清单
for _ in range(max_iter):
    dockerfile = docker_agent(ctx, feedback)
    eval_sh    = evalscript_agent(ctx, feedback)
    ok = rule_check(dockerfile, eval_sh)
    feedback = test_analysis_agent(logs, ok)
    if feedback.solved or feedback.unsolvable:
        break

四个智能体的分工

仓库探索：只开三个窄接口 browse/search/digest，成本感知地读 README/CONTRIBUTING/依赖清单/CI，把安装与测试线索喂给下游。

Dockerfile：拿探索结论造可复现镜像，迭代修依赖。

评测脚本：难点是精准锁定与 issue 相关的测试，必要时合成新用例；因引入新用例，SWE-Bench 的静态 fail2pass 脚本不再适用，改为从零写结构化 bash + OPENSWE_EXIT_CODE 标记。

测试分析：无论通过与否都复查——通过看是否有硬编码 exit code 作弊，失败则定位是 Dockerfile / 脚本 / 本质不可解，回传定向反馈或早停。

双向验证：只有"打 test-only patch 时测试 fail、打完整 fix patch 后测试 pass"两条都满足才接受，正则解析 OPENSWE_EXIT_CODE=X 作为最终信号。

3. 工程优化与分布式构建

规模化时真正致命的不是算法而是工程：网络抖动导致构建超时、镜像重复重建、僵尸容器耗尽磁盘、单点故障拖垮全局。论文用"预建基础镜像 + 本地仓库缓存 + 分层缓存 + 镜像复用"压掉构建成本，再用一套解耦、容错的数据并行框架撑起 64 节点集群，两周完成 45,320 环境。

# 环境构建侧四个提速点
base_images = prebuild(py="2.7, 3.5~3.14") # 预装 conda
                              # + 镜像源，免运行时装 Python
repo = COPY(local_bare_cache)   # 不在容器内 clone
                              # 去掉 API 限流/长 hash 出错
dockerfile = layer_aware(          # 稳定层靠前被缓存
    stable_first=True, deps_late=True)
image_reuse = keep_until(dockerfile_changes) # 只改
                              # 脚本时复用镜像 → 5× 提速

# 分布式: 64 节点数据并行
cluster = 64 ECS(cpu=32, ram="128G", ssd="4T")
queue   = FileMQ(shared_fs)   # 生产者/消费者解耦
proc    = systemd(auto_restart=True)
limit   = cgroup(cpu=4, mem="24G", disk="200G")

为什么这些细节重要

预建基础镜像：Python 2.7 及 3.5–3.14 全套预装 conda + 镜像源，直接消灭"运行时装依赖超时"这个最大失败源。

本地 bare 仓库 + COPY：把 GitHub API 限流和网络故障彻底移出 agent 循环，还避免智能体重复长 commit hash 出错。

分层缓存：稳定层靠前、依赖安装放后层，迭代改依赖时只重建廉价层。

镜像保留策略：只有 Dockerfile 变了才重建镜像，常见的"只改评测脚本"场景直接复用，≈5× 加速。

容错框架：刻意不用 MPI/Ray 这类紧耦合——数据并行 + 文件消息队列 + systemd 自愈 + cgroup 隔离 + 自动清理僵尸容器，任一节点挂掉不丢数据。每节点 32 vCPU/128G/4T SSD 是实验找到的近最优工作点。

4. 难度感知筛选与训练数据构造

光有规模不够。论文用轨迹通过率反推每个环境的难度：用 GLM-4.7 在每个环境上采样 4 次，只保留"4 次中对 1~2 次"的实例——全对说明太简单（无学习信号），全错说明不可解或标注有问题，恰好落在难度前沿的才最有训练价值。再做数据清洗与防作弊，最后用改造版 slime 做多轮 SFT。

# 难度 = 4 次采样的通过次数
traj = [rollout(env, model="GLM-4.7",
              temp=1.0, ctx="200k", steps=300)
        for _ in range(4)]
n_pass = sum(t.correct for t in traj)

if n_pass == 0:  drop("不可解/标注错")
elif n_pass >= 3: drop("太简单")
else:            keep(traj)   # 1~2/4 → 难度前沿

# 数据清洗 + 防 reward hacking
mask_steps(has_format_err | has_error_obs)
drop_if("git pull" in bash_action)

# SFT (改造 slime 支持多轮 action mask)
sft(base="Qwen2.5-32B/72B-Base",
    max_tok=128_000, epochs=5, bs=128,
    lr=cosine(1e-5, 1e-6))

关键设计

难度用通过率当难度信号：不需要额外打标，采样 4 次的"1~2 次对"天然筛掉不可解与平凡实例，把预算集中在学习效率最高的前沿。

防 reward hacking：删掉 bash 动作里含 git pull 的轨迹（会直接拉到正确代码），并对含格式错误/报错观测的步骤做 mask，只学干净动作。

多轮 action mask：改造 THUDM 的 slime 支持多轮训练时正确 mask，128k 上下文让长轨迹完整进 loss。

成本构成：$576K 的筛选成本主要是 GLM-4.7 在全量 OpenSWE+SWE-rebench 上"两个 scaffold × 各 4 次"采样的 API 费 + 每条轨迹的 Docker 执行费。

四个结论

两个规模都 SOTA：32B 62.4%（用非 Coder base 就超过最强 Coder 基线），72B 66.0%（比 daVinci-Dev-72B 高 7.5）。

大模型更吃数据：32B→72B 提升 3.6，而 daVinci-Dev 同条件只涨 2.4，说明高质量环境让大模型更能发挥参数量。

scaffold 无关：OpenHands 与 SWE-Agent 都受益，非绑定特定框架；SWE-Agent 稳定略高 1~3。

质量 > 来源：OpenSWE 单用即全面超 SWE-rebench（32B SWE-Agent +12.2）；72B 上两者混合最佳达 68.0，但 32B 混合反而略降（对分布漂移更敏感）。

6. 关键洞察

这篇工作的价值不在某个新模型或新算子，而在把"造 SWE 训练环境"这件事做成了可复现、可扩展、有质量把关的系统工程，并用实证澄清了"规模 vs 质量"的关系。

可借鉴的点

最核心难度感知筛选：把"通过率落在中间"当作数据价值的代理，是最省事又有效的课程/清洗信号，可直接搬到其它 RL/agent 数据构造。

工程规模化工程：预建镜像、本地仓库缓存、镜像复用 5×、文件消息队列容错——这些才是两周造 4.5 万环境的真正功臣。

生态全透明：Dockerfile / 评测脚本 / 分布式流水线全开源，把工业级 SWE 数据的门槛拉到学术组可及。

局限与延伸：仅限 Python、仅 SFT（未上 RL）、构建/采样模型（DeepSeek-V3.2 / GLM-4.7）本身能力决定环境与轨迹质量上限。log-linear 无饱和暗示继续扩数据仍有空间。可与 agentfounder.html、mirothinker.html 等 agent 训练笔记对照。
