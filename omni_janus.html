1. 核心理念 — 解耦视觉编码

# Janus 核心架构决策
# 理解和生成使用 不同 的视觉编码器
# 但共享 同一个 LLM backbone

# ===== Understanding Path =====
# 使用 SigLIP 连续视觉特征
vision_encoder = SigLIP("SO400M/14")
understanding_adaptor = MLP(

# 使用 VQ tokenizer 离散视觉 token
vq_tokenizer = VQTokenizer(
    codebook_size=16384,
    downsample=16
)
generation_head = Linear(
    in_features=2048,  # LLM hidden
    out_features=16384  # codebook
)

# ===== Shared LLM =====
llm = DeepSeekLLM(
    hidden_size=2048,  # 1.3B
    # or 4096 for 7B
)

# Key insight:
# Understanding: 需要高层语义表征
# Generation: 需要低层空间细节
# 一个编码器无法同时满足两者!

核心洞察：解耦视觉编码

问题：之前的统一模型（如 Chameleon）对理解和生成使用同一个视觉编码器，但这产生了表征冲突：

- 理解需要高层语义特征（压缩信息）

- 生成需要低层空间细节（保留重建信息）

Janus 方案：为两条路径各配最优编码器，但共享同一个 LLM backbone — 解耦编码，统一推理。

$\text{Enc}_{\text{understand}}(I) \neq \text{Enc}_{\text{generate}}(I)$
$\text{Understanding: SigLIP} \to \text{continuous}$
$\text{Generation: VQ} \to \text{discrete}$

2. 双路径架构详解

# ===== Understanding Pathway =====
# SigLIP-L/14 @ 384×384
# patch_size=14 → 384/14 = 27
# grid: 27×27 = 729 tokens

class UnderstandingEncoder:
    vision = SigLIP_L(
        input_size=384,
        patch_size=14,
        hidden_dim=1152,
        # output: [B, 729, 1152]
    )
    adaptor = Sequential(
        Linear(1152, 2048),
        GELU(),
        Linear(2048, 2048),
        # output: [B, 729, 2048]
    )
    # → concat with text tokens → LLM

# ===== Generation Pathway =====
# VQ Tokenizer: 16× downsampling
# 384/16 = 24 → 24×24 = 576 tokens

class GenerationDecoder:
    codebook_size = 16384
    downsample_ratio = 16
    # LLM autoregressively predicts
    # 576 discrete code indices
    # → VQ decoder → 384×384 image

    gen_head = Linear(
        2048, 16384  # → codebook logits
    )
    vq_decoder = VQDecoder(
        codebook=16384,
        output_size=384
    )

# ===== Shared LLM =====
# DeepSeek-LLM 1.3B or 7B
llm = DeepSeekLLM(
    vocab_size=100000,
    hidden_size=2048,  # 1.3B
    num_layers=24,
    num_heads=16,
)

关键参数

Understanding Encoder: SigLIP-L/14 @ 384x384, 产生 729 个 1152-d token (27x27 grid)，经 2-layer MLP 投影到 LLM 维度。

Generation Tokenizer: VQ codebook 16384, 16x downsampling. 384x384 image → 24x24 = 576 discrete tokens, LLM 自回归预测。

Shared LLM: DeepSeek-LLM 1.3B (h=2048, L=24) 或 7B (h=4096, L=30)。两条路径的所有 token 在同一个 transformer 中处理。

$\text{LLM}(\text{SigLIP}(I) \oplus \text{text}) \to \text{answer}$
$\text{LLM}(\text{prompt}) \to c_1, c_2, \ldots, c_N$
$\text{VQ-Dec}(c) \to I_{384 \times 384}$

3. 为什么解耦优于统一

# ===== 统一方法 (Chameleon / Emu3) =====
# 同一个 VQ tokenizer 编码所有视觉信息
class UnifiedModel:
    # ONE encoder for BOTH tasks
    visual_enc = VQTokenizer(codebook=8192)

    # 理解时: VQ tokens 丢失高层语义
    # → 需要 LLM 从离散 code 重建语义
    # → understanding 性能受限

    # 生成时: 如果用连续编码器...
    # → 没有离散 token 做 AR 生成
    # → generation 无法工作

    # 表征冲突！
    # understand: compress → semantics
    # generate: preserve → spatial

# ===== Janus 解耦方法 =====
class JanusModel:
    # 各取所长
    understand_enc = SigLIP()
    # → rich semantic features
    # → optimal for comprehension

    generate_enc = VQTokenizer()
    # → discrete codes for AR
    # → optimal for generation

    shared_llm = DeepSeekLLM()
    # → unified reasoning core
    # → no representation conflict

# ===== 结果 =====
# Janus-1.3B beats:
#   - LLaVA-1.5-7B on understanding
#   - DALL-E 2 on generation
# Despite being 5x smaller!

表征冲突的本质

统一编码器的困境：

理解任务要求编码器压缩视觉信息为高层语义 — 抛弃细节，保留含义。

生成任务要求编码器保留空间细节 — 每个像素位置的信息都要可重建。

这是根本性矛盾：压缩 vs 保留。

实验证据：

Janus-1.3B 在理解上超越 LLaVA-1.5-7B (5x 大)，在生成上超越 DALL-E 2 (5x 大)。解耦使得每条路径都能达到各自最优。

$\text{Understand: } \text{compress}(I) \to \text{semantics}$
$\text{Generate: } \text{preserve}(I) \to \text{spatial}$
$\text{Unified: } \text{compress} \neq \text{preserve}$
$\text{Janus: each path} \to \text{optimal encoder}$

4. Janus Pro — 生成质量飞跃

# Janus Pro 核心改进:
# 1. 更强的 VQ Tokenizer
# 2. 更大的 LLM backbone
# 3. 优化训练策略

# === 改进 1: VQ Tokenizer ===
# Janus 原版: VQ-16 (16x downsample)
#   384×384 → 24×24 = 576 tokens
#   codebook = 16384
# Janus Pro: 增大 codebook + 更好训练
#   codebook = 16384 → 同样大小
#   但训练时引入更强的 perceptual loss
#   + adversarial training (GAN loss)
#   → 重建质量大幅提升

tokenizer_pro = VQTokenizer(
    codebook_size=16384,
    spatial_downsample=16,
    encoder="improved CNN",
    losses=[
        "reconstruction (L1+L2)",
        "perceptual (LPIPS)",
        "adversarial (PatchGAN)",
        "codebook (commitment)",
    ],
)

# === 改进 2: 更大 LLM ===
# Janus: DeepSeek-LLM 1.3B / 7B
# Janus Pro: DeepSeek-LLM 1B / 7B
# 重点在 7B 版本的训练优化

# === 改进 3: 训练策略 ===
# Stage 1: 统一预训练
#   - 理解: image-text pairs (大规模)
#   - 生成: text-image pairs (大规模)
#   - 比例: 理解:生成 ≈ 7:3
# Stage 2: 监督微调
#   - 理解: VQA, 对话, OCR
#   - 生成: 高质量 text→image
# Stage 3: 生成增强
#   - 解锁 cfg (classifier-free guidance)
#   - 提升生成多样性和质量

# 推理时生成配置:
gen_config = {
    "cfg_weight": 5.0,
    "temperature": 1.0,
    "top_p": 0.95,
    "image_tokens": 576,
    "resolution": "384×384",
}

Janus Pro 三大改进

1. VQ Tokenizer 升级：加入 perceptual loss (LPIPS) + adversarial loss (PatchGAN)，重建从模糊变清晰。这是生成质量提升的最大来源。

2. 训练比例优化：理解:生成 = 7:3 的预训练配比，加入 Stage 3 专门强化生成（CFG guidance）。

3. 规模验证：7B 版本在 GenEval 上超越 DALL-E 3 (0.80 vs 0.67)，同时理解能力保持强劲 (MMBench 79.2)。

CFG (Classifier-Free Guidance)

推理时对同一 prompt 做两次 forward：一次有条件 (text prompt)，一次无条件。用两者的差值增强生成方向性：

$\hat{\epsilon} = \epsilon_{\text{uncond}} + w \cdot (\epsilon_{\text{cond}} - \epsilon_{\text{uncond}})$
$w = 5.0$ (guidance weight)

核心结论

Janus Pro 证明：解耦架构 + 强 VQ tokenizer + CFG 就能在统一模型中达到专用生成模型的水平，不需要 diffusion。

生成质量的瓶颈不在 LLM，而在 VQ tokenizer 的重建能力。
