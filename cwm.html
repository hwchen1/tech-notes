1. 概览:"代码世界模型"与神经调试器愿景

CWM 的核心不是又一个 coder 模型,而是一个研究命题:如果让模型在训练中观察代码怎样一步步执行(局部变量如何变化、函数如何返回、智能体如何在仓库里改代码跑测试),它是否能建立起对"计算环境动态"的内部世界模型,从而在推理时进行有据可依的、无需真实执行环境的代码推理?训练分三段:通用预训练 → 代码世界建模 mid-training(灌入执行轨迹与 agent 轨迹)→ 后训练(SFT + 多任务 RL)。

为什么叫"世界模型"

强化学习里 world model 指对环境状态转移 $s_{t+1}=f(s_t,a_t)$ 的可学习模拟器。CWM 把"代码执行环境"当作这个环境:状态=程序内存(局部变量),动作=被执行的那行代码,轨迹就是 observation-action 序列。

学会预测这种转移后,模型可以在"脑内"模拟执行——不必真的开解释器。

主张Neural debugger 愿景:CWM 想让 trace 预测超越传统调试器——常数时间跳到未来某行、反推能到达某状态的输入、学习程序状态的抽象表示以支撑验证/生成。当前论文只放出开放权重的研究基座(不是通用聊天模型,刻意省去 RLHF)。基座也是 ssr.html(Self-play SWE-RL)的起点。

2. 可执行仓库镜像 + Python 执行轨迹追踪

世界模型数据的前提是大规模可执行环境:把 GitHub 仓库封成能装好依赖、跑通测试的 Docker 镜像。人工搭不可扩展,于是双管齐下——RepoAgent(LLM 智能体读文档搭环境)+ Activ(复用 GitHub Actions CI,用 act 本地跑 workflow、注入 pytest fixture 抓取构建态并 commit 成镜像),共产出 >35k 镜像。有了镜像就能逐行追踪 Python 执行:每执行一行记录局部变量(状态)与该行代码(动作),编码成自定义 token 序列——这就是"神经代码解释器"数据。

# CWM 执行轨迹的 token 格式(图3)
def f(a, b):
    y = a
    for i in range(b): y += y*i
    return y
def main():  # << START_OF_TRACE
    return f(1,3)
# ── 模型预测的 observation-action 流 ──
<|call_sep|>{}<|action_sep|> def main():
<|line_sep|>{}<|action_sep|> return f(1,3)
<|call_sep|>{"a":"1","b":"3"}<|action_sep|> def f(a,b):
<|line_sep|>{"a":"..","y":"1"}<|action_sep|> for i in..
   # 未变的变量用 ".." 省略, 仅记增量
<|return_sep|><|action_sep|> return y
<|arg_sep|>"6"<|frame_sep|>   # 返回值=6

格式设计要点

状态 = JSON 局部变量字典,动作 = 单行代码;未变量用 .. 省略只记增量,大幅压缩轨迹长度。

四类来源:① 函数级(在线抓函数 + fuzzing/Llama3 造 IO,120M 条);② CodeContests 解答(33k 片段/70k 轨迹);③ 仓库级(21k 镜像单测 + git 历史 commit,~70k);④ 自然语言轨迹(Qwen3-32B 把结构化轨迹改写成自然语言 75M+110k,便于向推理迁移)。

数据RepoAgent vs Activ:前者靠 LLM 读人类文档搭环境(易受文档失真影响);后者巧妙复用 CI——"机器指令必须准确否则 CI 直接报错",可靠性更高。二者并行产出 >35k 可执行镜像,是全部世界模型数据的地基。

3. ForagerAgent:智能体中训练数据

第二大世界模型数据是 agent 在真实仓库里改代码的多步轨迹。ForagerAgent(用 Llama3-70B 或 Qwen3-235B 驱动)在可执行镜像里执行 SWE-Agent 工具集(create/edit/bash/view),产出两类任务:mutate-fix(用 AST 变异合成 bug 再让 agent 修)和 issue-fix(真实 GitHub issue+PR)。关键理念:目标是学"与代码环境交互的世界模型",所以不按成败过滤轨迹——失败轨迹同样是环境动态的合法样本。共 3M 轨迹、10.2k 镜像、3.15k 仓库,issue/mutate ≈ 55/45。

# mutate-fix: AST 变异合成可验证 bug
mutations = [
  "remove function / part",      # 7%
  "drop / reorder arguments",      # 9%
  "swap two variables",            # 6%
  "remove import / return",        # 11%
  "replace operator",              # 12%
]
assert tests_pass(orig)          # 变异前全绿
m = apply_ast_mutation(func)
assert not tests_pass(m)         # 变异后必红

# agent 收集轨迹(不按成败过滤!)
traj = forage(agent, env, task=m)
if jaccard_minhash(traj) < 0.5:  # 近似去重
    keep(traj)
mask_loss(observations, p=0.5)   # 观测多样性低,半数不算loss

与后训练的关系

预先埋下 agent 先验:mid-training 就大量见过"在 Docker 里读文件、跑测试、改代码"的完整交互,使后续 SWE RL 更容易起步。

同时学 agent 与环境两侧:模型不仅学动作,也学预测环境返回(observation),这正是"世界模型"——但因环境观测多样性低,对 50% 观测 mask loss。

工程可验证性保证:mutate-fix 用 AST 解析先滤掉无法施加的变异,再验证"变异确实让测试变红";issue-fix 验证"PR 前测试红、PR 后转绿且必要充分"。防污染:剔除所有 SWE-bench 用到的仓库及其 fork。变异分布见左侧百分比。

4. 架构 · 两阶段预训练 · Scaling Laws

CWM 选 32B dense(非 MoE),理由是"下游开源研究易用"。注意力用 局部/全局 3:1 交替的滑动窗口(局部 8192、全局 131072),既省长上下文算力又保留全局能力;GQA 48Q/8KV、SwiGLU、RMSNorm 前置、Scaled RoPE($\theta=10^6$,scale 16,从 mid-training 起)。预训练两阶段共享调度器/优化器态但换数据与长度:通用 8T(8k 上下文)→ 世界建模 mid-training 5T(131k 上下文、batch 增到 33M token)。整个 cosine 衰减按 13T 规划,最后 5T 归 mid-training。

# FLOP/token 显式含注意力项 (式1)
$M = \underbrace{6 N_{\text{ne}}}_{\text{linear}}
     + \underbrace{6\,d L S}_{\text{attention}}$

# 固定 model-to-data 比 (贴合最终 run)
$D / M = 40$   # ≈ Chinchilla 的 8× 数据

# 8 个算力档 (2e18~2e20 FLOP) 拟合
$LR(C) = 19.29 \cdot C^{-0.177}$
$BS(C) = 30.17 \cdot C^{\,0.231}$

# 长上下文 mid-training: 按长度分桶
buckets = [(0,16385], (16385,65537],
           (65537, INF)]
# 同 step 所有 worker 抽同桶 → 免长短轨迹互等
# 中桶限全局注意力 ≤ 32768 再提速

关键工程取舍

长上下文mid-training 有 ~30% 文档 >65k token,故直接把最大长度设成 131k,省掉专门的长上下文 finetune 阶段。为避免"短文档 worker 空等长文档 worker",按长度分桶采样,并对中桶限全局注意力到 32768。

大 batch 治长上下文:小 batch 训长文档效果差,推测因单 batch 文档数太少、梯度方差大 → mid-training 把 batch 提到 33M token。

scaling贴合最终 run 的 scaling law:遵循 Gadre 等的思路,固定 $D/M=40$(约 Chinchilla 8× 数据,偏 inference-optimal),而非追 Chinchilla 计算最优,直接外推出 32B/8T 的 LR、BS。FLOP 公式显式含注意力项,对小模型/长上下文更准。

5. 后训练:SFT + GRPO 变体 + 四类 RL 环境 + 联合 RL

后训练先 SFT(100B token,引入 <|reasoning_thinking_start/end|> 包裹推理、默认非推理模式可注入激活),再做多任务多轮异步 RL。刻意不做 RLHF(不追求通用聊天)。RL 用 GRPO 变体并做了一串修正:多轮 mask、用回报 $R_i$ 而非奖励、去 $\sigma$ 归一化(避免难度偏置)、去长度归一化(除以 $N=131072$)、clip-higher($\varepsilon_{\text{high}}=0.25$)、无 KL、跳过零优势/陈旧轨迹、长度加权均值、乱码检测。四类环境:agentic SWE、代码、agentic 代码、数学,联合训练。

# GRPO 变体的优势(去 σ / 去长度归一)
$\hat{A}_i = R_i - \mu$     # μ=长度加权均值
# 而非 GRPO 的 (r_i-μ)/σ,避免难度偏置

# SWE RL 混合奖励(隐藏测试 + 补丁相似度)
def reward_swe(patch, tests, oracle):
    if all_pass(tests):  return 1.0
    sim = similarity(patch, oracle)
    return 0.0 if sim > 0.5 else -1.0
    # 阈值化(非连续)更稳,难例仍有信号

# 自举: RL↔拒绝采样↔SFT 三轮
# SWE-bench pass@1: 30% → 37% → 43%

# 联合 RL 采样比 · 三阶段课程
mix = {"SWE":.4, "code":.4, "math":.2}
rehearsal = 1/3          # NLL loss ×0.1
filter = solve_rate in [0.1, 0.7]  # 最大化信号

四类环境

Agentic SWE:端到端解 issue,极简工具(bash 为核 + edit/create/submit 插件),最多 128 轮/131k;混合奖励;数据自举 3 轮解决"格式陌生"冷启动,最终 12.6k 实例。

代码 / agentic 代码:竞赛编程,±1 奖励(需恰一个 </think>+一个代码块+过全部单测);多语言。数学:LaTeX box 答案,启发式验证器,278k 题。

RL联合 RL 三阶段课程:阶段1 格式自举 + action-length 奖励调度(软最大 8k→硬 64k);阶段2 增代码占比到 50%、加 Rust/Go/Java/JS、50% 概率禁插件逼用原生命令;后期按 solve-rate 分桶、难例反比加权。异步 RL:worker 边生成边接收新权重(可在生成中途换权重),用 moodist 直传、FastGen 推理,GPU 几乎不空转。

6. 实验结果与"世界模型"独有评测

CWM 在 SWE-bench Verified 达 65.8%(best@16 test-time scaling)/ 53.9%(单次),同规模开源模型 SOTA,逼近更大/闭源模型。更有意思的是论文专门验证"世界模型"能力:执行轨迹预测、程序终止预测(HaltEval)、算法复杂度预测(BigOBench)——这些是普通 coder 模型不评的维度。

世界模型能力的证据

轨迹执行轨迹预测:CruxEval-O 上"full trace"88% 逼近语言推理 94%,但 token 少一半(497 vs 1164);状态/动作精确匹配 >96%、JSON 格式 100%——模型真的学会了"脑内跑代码"。

终止程序终止预测(不可判定问题):HaltEval 上直接预测仅 0.37,但开推理后飙到 0.94——出乎作者意料,提示 reasoning + 世界模型能逼近"考察多个具体输入再泛化"的判断。

复杂度BigOBench:时间复杂度预测 all@1 41.3(超 Qwen3-32B/coder-30B/Gemma-3-27B),时间复杂度生成全指标最佳;去掉复杂度约束时 code-only pass@1 掉得最少,说明能兼顾主任务与附加约束。空间复杂度略逊 Qwen3-32B。
