2. 固定状态推理

# RecurrentGemma 推理: 恒定显存

class RecurrentGemmaInference:
    def __init__(self, model):
        self.model = model
        # 推理状态 (固定大小!)
        self.states = []
        for layer in model.layers:
            if layer.type == 'rglru':
                # RG-LRU: d 维向量
                self.states.append(zeros(d))
            else:
                # Local Attn: 固定 window KV
                self.states.append(
                    RingBuffer(window_size=2048)
                )

    def generate_one_token(self, token):
        x = self.model.embed(token)
        for i, layer in enumerate(self.model.layers):
            x = layer.forward_one_step(x, self.states[i])
        return self.model.head(x)

# 显存分析:
# RG-LRU 层 (13层): 13 × d = 13 × 2560 = 33K params
# Local Attn 层 (13层): 13 × W × 2 × d
#   = 13 × 2048 × 2 × 2560 = 136M params
# Total state: ~136M (固定, 不随序列增长!)

# 对比 Gemma-2B Transformer:
# KV cache (seq=32K): 26 × 32K × 2 × 2560 = 4.3B params
# → RecurrentGemma 状态比 Transformer 小 30×!

恒定显存的魔力

RecurrentGemma 的推理状态：

$\text{State} = \underbrace{13 \times d}_{\text{RG-LRU}} + \underbrace{13 \times W \times 2d}_{\text{Local KV}}$ $= O(L \cdot W \cdot d) = \text{常数}$

不管处理 4K 还是 1M tokens，推理显存都不变！

定位总结

RecurrentGemma 不是为了"替代" Gemma，而是为不同部署约束提供选择：

$\text{质量} \approx \text{Gemma} - 1\%$ $\text{推理效率} \approx 2\text{-}3\times \text{ Gemma}$ $\text{内存} = O(1) \text{ vs } O(N)$
适用判断：如果你的场景是"固定设备 + 长对话 + 吞吐优先"，选 RecurrentGemma。如果是"精确 RAG + 短上下文 + 质量优先"，选 Gemma。两者可以用完全相同的 SFT/RLHF pipeline。

开源可用：google/recurrentgemma-2b 和 google/recurrentgemma-9b 均在 HuggingFace 开放权重。
