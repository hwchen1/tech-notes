1. 病根与解法 — 把昂贵的 search-browse 循环搬到离线

训练 deep research agent 需要长程轨迹(交织 search / 证据聚合 / 多步推理)。但现有数据流水线几乎都依赖商用 web API(Google Search 等),带来三个致命问题:① 贵——每条失败的搜索路径也照样烧 API 钱;② 不稳定不可复现——活的 web 一直在变,同一流水线跑两次结果不同;③ 难分析——内部 search 事件依赖变动环境,benchmark 也不暴露稳定的 gold-document 标注,无法精确分析"证据何时被检索/打开/错过"。

解法两个核心想法:① 解耦语料构建与轨迹生成——做一次性在线 bootstrap 收集 answer-supporting 文档、建离线语料+搜索引擎,然后整个多轮合成循环全程离线;② 用三个最小浏览原语 search / open / find 显式建模浏览行为,让 teacher 不只学"检索什么",还学"怎么查看文档、怎么定位证据"。结果:成本归零($0 vs Serper $5,760)、完全确定可复现、无速率限制可大规模并行。

# OpenResearcher: 离线轨迹合成三阶段

# ===== ① QA 收集 =====
# 选 MiroVerse-v0.1 (需长程多跳的难题)
qa = sample(MiroVerse, 10%)   # ~6K QA
qa = normalize_answer(qa)     # 简洁可验证
# 原始轨迹不可用 → 全部从头重新生成

# ===== ② 离线语料 (一次性在线 bootstrap) =====
for q in qa:                  # answer-guided 检索
    query = q.question + q.answer  # 拼答案提召回
    gold += serper_search(query)   # 仅此处用在线 API
gold = clean_dedup(gold)      # 10K gold docs
corpus = gold + fineweb_15M   # 加 15M 干扰文档(~10T tok)
index  = FAISS(Qwen3_Embed_8B(corpus))

# ===== ③ 离线轨迹合成 =====
# GPT-OSS-120B teacher, 不给参考答案
traj = teacher.solve(q, tools=[search, open, find],
                     engine=offline_index)
# 过滤: 超长/malformed/未出答案 → 97K+ 轨迹

关键 insight

解耦是关键: 把"建库"(一次性、需在线)和"合成轨迹"(海量、全离线)拆开。在线 API 只在 bootstrap 阶段碰一次收集 gold docs,之后所有多轮搜索都打本地 FAISS → 成本/可复现/可分析全部解决。
gold + 干扰共存: 语料 = 10K gold docs(保证证据可检索)+ 15M FineWeb(当干扰项,模拟真实 web 的噪声与歧义)。这样离线环境既"答得出"又"不简单"。
从头重生轨迹: MiroVerse 自带的部分轨迹不可直接监督(证据未必支撑答案、trace 退化)。只用干净 QA 对,让 teacher 不看答案从头 search 出来 → 才是真实长程行为。
成本对比震撼: 合成 97K 轨迹含 5.76M 次搜索,Serper 要 $5,760、SerpAPI 要 $28,800,离线检索器 $0。这才让"大规模长程合成"在全开源下首次成为可能。

2. 三浏览原语 search / open / find — 从"检索"到"真浏览"

大多数 agentic search 系统把 search 当成"发一个 query、拿几条 snippet、直接在上面推理"——这种抽象在 deep research 上撑不住,也和人类真实研究方式相差甚远(人会:发广查 → 打开有希望的页 → 略读定位关键段落 → 据部分证据改写 query 再迭代)。OpenResearcher 用三个最小原语显式建模这套流程,逐级把焦点从"语料"收窄到"文档"再到"证据":search(返回 top-K 标题/URL/snippet,广覆盖)→ open(取整页全文,模拟点进网页)→ find(页内精确字符串匹配,做实体查找/事实核验)。

消融 (RQ4) 强力证明三原语缺一不可:仅 search 准确率仅 43.86%、gold 命中 1.45%;+open 跃升到 56.39%(证据访问质变);三原语全开 达 62.17%,gold 命中升到 53.37%,且首次命中更早(20.6→17.2 轮)、tool call 和 token 双降。

# 三原语: 多尺度证据发现

# search — 语料级广覆盖
results = search(query, top_k=K)
# → [(title, url, snippet), ...]
# 识别候选源, 但只有 snippet (不全)

# open — 文档级精读
page = open(url)
# → 整页全文, 模拟点进网页
# 关键: 把 gold 从 "搜到" 变成 "打开"

# find — 证据级定位
hits = find(page, "MIT")
# → 页内精确字符串匹配
# 实体查找 / 事实核验 / 假设 grounding

# 焦点逐级收窄:
#   corpus → document → evidence

# 合成约束:
#   只用这3个工具 · 每次 tool call 前先推理
#   不给参考答案 · 自信才终止

原语关键点

search ≠ deep research: 只发 query 拿 snippet 是浅层抽象。真实研究是"广查 → 打开 → 定位 → 迭代"的多尺度过程,三原语正是把这个过程显式化。
open 是质变点: 仅 search 时 gold 命中只有 1.45%——snippet 根本不够。加上 open 后 gold 命中飙到 51%,准确率 +12.5 点。RQ5 进一步证明:P(对|打开 gold)=86.7% 远高于 P(对|搜到 gold)=61.8%,"搜到"不等于"看到"。
find 锦上添花: 即便检索已成功,页内精确定位仍能再提准确率、并让首次命中更早、token 更省。证据 localization 本身有独立价值。
逐级收窄: corpus → document → evidence 三级漏斗,对应三原语。不同工具组合 = 不同信息发现尺度,这是 deep research agent 设计的核心抽象。

3. 受控分析与结果 — 离线训练竟泛化到在线 web

因为环境完全离线且全程可观测(语料/检索后端/浏览动作全固定),OpenResearcher 能做活 web 上做不了的受控分析,回答 5 个 deep research 流水线设计的关键问题 (RQ1-5)。最反直觉的结论:仅在离线合成轨迹上 SFT(无 RL、无任何在线训练),不仅在离线 BrowseComp-Plus 上从 20.8%→54.8%(+34.0,超 GPT-4.1/Claude-4-Opus),还泛化到活 web benchmark(BrowseComp 26.3 / GAIA 64.1 / xbench 65.0),全面碾压同类开源 agent(ASearcher 5.2/52.8/42.0)。训练用 Megatron-LM、8×H100 约 8 小时、256K ctx 预打包、347 步、GBS=64。

# RQ1-5: 受控分析关键发现

# RQ1: 答案对错是必要的过滤信号吗? → 不是!
correct_only:  54.81   # 三者几乎相同
incorrect_only:55.06   # 失败轨迹照样有监督价值
all_traj:      54.46   # (search结构/工具序/停止行为)

# RQ2: 一次性在线 bootstrap 必要吗? → 必要!
with_gold:    gold_hit 29.5  BC+ 54.81
without_gold: gold_hit  1.7  BC+  6.35  # 崩!

# RQ3: 多少 turn 预算够? → ~100 轮后 plateau

# RQ4: 浏览工具重要吗? → search<+open<+find
#   43.9% → 56.4% → 62.2%

# RQ5: 搜到 gold 就答对吗? → 不!打开才行
P(correct | search_hit) = 61.8%
P(correct | open_hit)   = 86.7%
P(search_hit | correct) = 99.4%  # 证据暴露≈必要

# 失败轨迹 tool call 近2倍 (71.7 vs 38.4)
# → 失败源于"搜索漂移"而非探索不足

分析关键点

RQ1 — 失败轨迹也有用: correct-only / incorrect-only / all 三种过滤下游准确率几乎一样(54.8/55.1/54.5,差<0.6 点)。监督信号其实藏在 search 结构、工具调用顺序、证据查看、停止行为里,而非答案对错本身。
RQ2 — bootstrap 不可省: 去掉 gold docs,gold 命中 29.5%→1.7%、轨迹准确率 56.9%→43.8%、下游 BC+ 54.8%→6.35% 直接崩盘。语料覆盖是离线合成的硬前提。
RQ5 — 搜到 ≠ 答对: P(对|搜到)=61.8% 但 P(对|打开)=86.7%;同时 P(搜到|对)=99.4%。即证据暴露几乎是必要条件,但不充分——这把"检索失败"和"推理失败"清晰区分开。
失败 = 搜索漂移: 失败轨迹 tool call 近 2 倍(71.7 vs 38.4),且多出来的主要是 search(48.7 vs 22.1)而非 open/find。说明难题的瓶颈是query 没问对、反复重写,不是步数不够——这正是设计显式浏览原语的动机。

4. 一句话总结

OpenResearcher 是第一个全开源、可复现的 deep research 轨迹合成流水线:用"一次性在线 bootstrap 建库 + 离线多轮合成"的解耦设计,把 search-browse 循环成本压到 $0,配合 search/open/find 三原语显式建模真实浏览,用 GPT-OSS-120B 合成 97K+ 长程轨迹。仅靠这些离线轨迹 SFT 一个 30B-A3B,就能在离线和活 web 两端都媲美甚至超越商用 deep research 系统。它和 AgentFounder(全离线合成 agent 数据)、MiroThinker(其 MiroVerse 正是本文 QA 来源)同属"用合成数据扩 deep research agent"这条主线,但独特点在于把整个搜索环境也开源化、离线化、可分析化。
