补丁 → 长程 → 自主构建

SWE 智能体的任务 Horizon 演进与未来 · 综述笔记 (2026-07) | 以"任务时间跨度"为主轴, 串起三个阶段: ① 补丁生成(SWE-bench Verified, 单文件条件采样) → ② 长程工程(SWE-Bench Pro / DeepSWE / Terminal-Bench, 企业级闭环) → ③ 自主构建(SSR / Agent0 自博弈, 从零造软件). 核心论点: 从补丁到构建不是"更大的补丁", 而是 conditional generation → closed-loop control 的范式切换, 需要 长上下文 + 多轮工具 + 分层规划 + 自我验证 四件套, 并以"代码世界模型"为底座. 末章展望自主构建之后: 软件工厂 / 持续自演化 / human-in-the-loop 边界 / 经济与对齐. 交叉链接 cwm.html · ssr.html · davinci_env.html

1. 主轴:任务 Horizon 决定一切

看 SWE 智能体的演进,最有解释力的不是"模型多大"或"分数多高",而是任务的时间跨度 (task horizon)——完成一个任务需要多少步自主决策、维持多久的状态一致性。Horizon 一拉长,能力要求就发生质变。整条演进线可以压缩成一根轴:补丁 → 长程 → 自主构建。

# 用 horizon 定义三个阶段
def stage(H):            # H = 期望自主决策步数
    if H < 1e2:   return "补丁生成"    # 定位+局部手术
    if H < 1e3:   return "长程工程"    # 跨模块+环境闭环
    return "自主构建"                # 需求→架构→交付

# 每阶段的代表评测 & 任务形态
stages = {
  "补丁": ("SWE-bench Verified", "给 issue 改几个文件"),
  "长程": ("SWE-Bench Pro / DeepSWE", "企业级/数天量级"),
  "构建": ("SSR / Agent0 自博弈",   "从零造软件"),
}

# 判据: 谁来决定"下一步做什么 / 做完没"
who_decides_next_step = "外部 oracle"   # 补丁
who_decides_done      = "agent 自己"    # 构建 ← 分水岭

为什么用 horizon 当轴

Horizon 是压力测试的真正变量:同一个模型,horizon 越长,越暴露"状态漂移、目标遗忘、错误累积"这些补丁任务里根本不出现的问题。分数只是果,horizon 才是因。

阶梯而非连续:每跨一档,主要瓶颈整体换一次——从"会不会改"到"撑不撑得住"再到"知不知道要做什么"。

核心判据一个任务能被"生成→提交→外部 oracle 判分"一次闭合,就还是补丁范式;一旦需要 agent 自己决定下一步、自己判断是否做完,就跨进了 agentic 自主构建。本笔记后续都围绕这条判据展开。

2. 阶段一 · 补丁生成:条件采样

最早的 SWE 能力本质是条件生成:给定 issue 和一份静态仓库快照,生成一段 patch 使隐藏测试通过。代表评测是 SWE-bench Verified(500 条人工核验、可解的单/少文件任务)。这里 repo 是只读输入,agent 不改变世界、只产出一次答案,判分完全外包给 fail-to-pass 测试。

# 补丁 = 一次条件采样 (世界是静态的)
patch ~ p(patch | issue, repo_snapshot)

# 典型 Agentless / 定点手术流程
files  = localize(issue, repo)      # ① 定位
edit   = generate_edit(files, issue)  # ② 局部修改
result = run_hidden_tests(edit)     # ③ 外部 oracle
# 无第 ④ 步: agent 不判断"做完没"

# 评分 (外部给定, 无需自我验证)
resolved = fail2pass and not pass2fail

# 能力上限: "看懂已有代码库并精确改一处"
bottleneck = ["定位精度", "局部编辑正确性"]

补丁范式的特征

世界静态:repo 快照不因 agent 动作改变,不需要预测"我这么改会发生什么"。

oracle 外包:done 的定义由隐藏测试给,agent 无需自己判断完成度。

分数已高但天花板明确:davinci_env.html 的 OpenSWE-72B 达 66.0、cwm.html 的 CWM 达 65.8——但这些数字衡量的都是"定点手术"能力。

为什么它不够:补丁范式假设"任务已被拆成一个可测的小改动"。真实工程里,"拆解"本身才是难的部分——而这正是下一阶段要处理的。

3. 阶段二 · 长程工程:闭环控制

当任务变成"跨模块、跨服务、需要真实环境交互、专业工程师要数小时到数天完成",条件生成就失效了。此时是闭环控制:每一步动作 $a_t=\pi(s_t)$ 都改变世界状态,而下一步基于被自己改过的状态 $s_{t+1}=f(s_t,a_t)$。代表评测:SWE-Bench Pro(长程 + 抗污染)、DeepSWE(113 个原创长程任务)、Terminal-Bench(完整终端环境)。

# 长程 = 闭环控制 (状态被自己改变)
s = env.reset(task)
for t in range(300):        # 数百步量级
    a = policy(s, memory)        # bash/edit/create/test
    s = env.step(a)           # 世界状态更新
    memory.append(a, s.obs)   # 必须记住 200 步前的决策
    if policy.thinks_done(s): break

# 致命点: 第 50 步工具调用失败没 recover
# → 后续 250 步全建立在错误状态上
robustness = P(tool_ok) ** horizon   # 指数衰减

# 落差量化 (CWM): 补丁强 ≠ 长程强
swebench_verified = 65.8
terminal_bench    = 26.3   # 撑不住长状态

范式已经切换

补丁 → 长程 不是量变:$p(\text{patch}\mid\text{issue})$ 的一次采样,变成 $\prod_t \pi(a_t\mid s_t)$ 的长序列决策。前者像"阅读理解",后者像"项目管理"。

鲁棒性指数级要求:补丁里工具错了大不了废掉这条;长程里第 50 步没 recover,后面全崩。所以长程 agent 对错误恢复的要求指数于补丁。

证据cwm.html 的 CWM 在 SWE-bench Verified 有 65.8、Terminal-Bench 却只有 26.3。这个落差量化了"补丁强 ≠ 长程强"——不是不会改代码,是撑不住那么长的状态一致性。SWE-Bench Pro / DeepSWE 用抗污染的原创长任务,正是为了逼出这道鸿沟。

4. 阶段三 · 自主构建:自定目标 + 世界模型

终点是"从零自主构建软件":没有现成 issue、没有现成测试当 oracle。agent 必须自己生成验收标准(写测试、定义 done 的条件),再自己去满足。两个关键技术底座:① 自博弈(SSR / Agent0)——自己出题自己判分,把训练从人工标注 issue 解放出来;② 代码世界模型(CWM)——能在脑内 rollout "这么改、测试会不会挂",把补丁生成升级成可规划的闭环。

# 自博弈: 自己定义任务与 oracle (SSR / Agent0)
while training:
    task  = proposer(base)      # 出越来越难的题
    spec  = write_tests(task)    # 自己造验收标准 ← 关键
    traj  = solver(task, env)
    R     = verify(traj, spec)   # 自己判分, 无人工
    base  = RL_update(base, R)
# 自举: 30% → 37% → 43% (SSR 观测)

# 世界模型: 规划前先预测环境反馈 (CWM)
def plan(goal, s):
    for a in candidate_actions(s):
        s_hat = world_model(s, a)   # s' = f(s,a), 想象
        if reaches(s_hat, goal): return a
    # 无需真跑就能剪枝 → 长程规划可行

两个底座缺一不可

自主度自博弈解放数据:ssr.html 的 30%→37%→43% 自举曲线,本质是"自主度"被 RL 一点点抬高;agent0(arXiv 2511.16043)用 curriculum/executor 双 agent 从零数据共演化,同一思路。

规划世界模型使规划可行:cwm.html 押注 $s_{t+1}=f(s_t,a_t)$,让 agent 不真跑就能预测反馈、剪枝搜索——这是长程规划的前提。

分水岭:greenfield 没有外部 oracle,"写测试定义 done" 从工具变成核心能力。谁能自己出题、自己判分、自己收敛,谁才真正跨过"自主构建"这道线。

5. 拉长 Horizon 硬性要求的四件套

从补丁走到自主构建,不是把模型放大,而是要同时补齐四项能力。任何一项跟不上,horizon 就在那里断掉。

能力补丁生成长程工程自主构建
长上下文相关文件即可 (~8k)跨模块 + 历史决策 (128k)128k+ 外加外部记忆/工作区
多轮工具可选, 错了废掉重来必须, 错误恢复关键必须, 长链路零崩 + 环境自建
分层规划几乎不需要子任务分解需求→架构→模块的多层规划
自我验证外部 oracle 代劳部分自查 + 外部测试自己写测试定义 done ← 核心
# ① 长上下文 ≠ 塞更多代码, 而是不失忆
#   补丁: ctx ≈ 相关文件
#   长程: ctx ≈ "我 200 步前为什么选这架构"
external_memory = TODO_md + README + scratchpad
# 突破单次上下文天花板的关键

# ② 多轮工具: 鲁棒性是长程地基
success = P(tool_ok) ** H     # H 越大越脆
need: retry / rollback / 环境自建 (RepoAgent)

# ③ 分层规划: 目标栈
plan = decompose(goal) -> [sub1, sub2, ...]
while stack: execute(stack.pop()); replan_if_drift()

# ④ 自我验证: greenfield 无 oracle
spec = write_acceptance_tests(goal)   # 自造标尺
done = all(run(spec))                 # 自己判完成

关键澄清

长上下文不是终点:纯靠 context window 撑长程有天花板,真正的长程 agent 靠外部记忆(把决策写进文件/TODO)突破单次上下文。

为什么是 agentic 而非补丁生成:补丁只需一次条件采样;这四件套合起来描述的是一个"感知-规划-行动-自查"的持续控制回路——本质是 agent,不是 generator。

外溢证据davinci_env.html:纯 SWE 训练带来 MATH-500 +12.2 / MMLU-Pro +8.9 跨域正迁移,而事实召回(MMLU/TriviaQA)几乎不动。说明这四件套训练的是通用的多步问题求解回路,不是窄代码技能——这也是"code agent 外溢成 general agent"的根据。

6. 自主构建之后:未来发展

当"从零造一个软件"成为常规能力,竞争焦点会再次上移。下面五条是我认为的下一段主线——从单 agent 造软件,到组织级、自演化、有边界、可对齐的软件生产系统。

# ① 单 agent → 多 agent 软件工厂
factory = Org(
    pm=planner, archs=[architect],
    devs=[coder]*N, qa=[tester], sre=[ops])
# 关注: 角色分工 / 通信协议 / 冲突消解 / 成本

# ② 持续自演化 (不再一次性交付)
while product.alive:               # 长期运维
    telemetry = monitor(prod)
    task = self_propose(telemetry) # 自己发现要做什么
    ship(solve(task)); learn(outcome)  # 在线学习

# ③ human-in-the-loop 边界 (越强越要停)
if action in {pay, deploy_prod, delete_data,
              sign, change_auth}:
    stage_for_human_approval()     # money-movement guard

# ④ 可验证性 = 增长瓶颈
throughput ∝ 1 / cost(verify)      # 判分比生成更贵

五条主线

1多 agent 软件工厂:单体造软件 → 拟组织的角色分工(PM/架构/开发/QA/SRE),难点从"会不会写"变成"通信协议、冲突消解、成本/延迟均衡"。

2持续自演化:交付不再是终点。agent 长期盯着线上遥测,自己发现该做什么、在线学习——horizon 从"造一次"趋于无穷("运营一个产品")。

3人机边界:能力越强,"何时停手交人"越是核心。Construction-Finance(arXiv 2606.22000)的 money-movement guard 就是范例:付款/部署/删数据/改权限等不可逆动作必须暂停待批。

4·5可验证性成为增长瓶颈:当生成几乎免费,吞吐由"判分成本"决定——自动化验证/形式化规约会成为新战场。经济与对齐:代码责任归属、供应链安全、对"自造软件"的审计与红队,将和能力本身一样重要。
