突破 LLM 输出长度瓶颈: 从 2K 到 10K+ tokens 连贯生成 | 2024-2025

1. LongWriter — AgentWrite 数据合成 + SFT (清华 GLM, 2024)

# ====== 核心问题 ======
# 即使模型能处理 128K 输入上下文,
# 输出通常限制在 ~2K tokens
# 原因: 训练数据中几乎没有长输出样本
# → 模型学到的"正常输出"就是短回复
#
# ====== AgentWrite Pipeline ======
# 核心思路: 用 agent 分步生成长文本
# 再用生成的长文本做 SFT 训练数据

# Step 1: Plan (生成大纲)
def agent_plan(task, target_length):
    # 输入: 写作任务 + 目标长度
    # 输出: 分段大纲 (section titles + lengths)
    outline = llm.generate(
        f"Plan a {target_length}-token article:\n"
        f"Task: {task}\n"
        f"Output: numbered sections with word counts"
    )
    return parse_outline(outline)

# Step 2: Write (逐段生成)
def agent_write(outline):
    sections = []
    for section in outline:
        # 每段 200-1000 tokens (模型擅长的范围)
        text = llm.generate(
            f"Write section: {section.title}\n"
            f"Length: {section.target_len} words\n"
            f"Context: {sections[-1] if sections else ''}"
        )
        sections.append(text)
    return sections

# Step 3: Merge (合并 + 连贯性修正)
def agent_merge(sections):
    full_text = "\n\n".join(sections)
    # 修正段间过渡和重复
    return llm.refine(full_text)

# ====== 训练流程 ======
# 1. AgentWrite 生成 6000 条长输出数据
#    (LongWriter-6K, 每条 2K-10K+ tokens)
# 2. 用 LongWriter-6K 对基座模型做 SFT
# 3. SFT 后模型直接生成长文本
#    (不再需要 agent pipeline)
#
# 开源: GLM-4-9B-LongWriter
# 评测: LongBench-Write
# 效果: 支持 10K+ token 连贯输出

AgentWrite 将长文本任务分解为 plan-write-merge 三步, 生成的长输出直接作为 SFT 训练数据

Key Insights

Root Cause: 输出长度瓶颈不是模型能力问题, 而是训练数据偏差 -- 几乎所有 SFT 数据都是短回复 (<2K tokens)
AgentWrite 核心: 把"一次性写长文"转化为"分段写短文再合并", 绕过单次生成长度限制
数据效率: 仅 6000 条 AgentWrite 生成的样本就能解锁 10K+ 输出能力, 说明模型本身有长输出潜力
Self-Bootstrapping: 用当前模型的 agent 模式生成数据 → SFT 后模型不再需要 agent → 蒸馏了 agent 能力到单次生成

LongBench-Write 评测

输出质量评分 + 长度达标率

GLM-4-9B-LongWriter 在 10K 目标长度下达标率 >85%, 对比 GPT-4 仅 ~50%

2. LongGenBench — 长输出质量评测基准

# ====== LongGenBench 评测维度 ======
# 专门评测 LLM 长输出质量的 benchmark
# (区别于长输入理解评测如 RULER/HELMET)

# 维度 1: Length Compliance
# 输出是否达到要求长度?
# metric: |actual_len - target_len| / target_len
# 失败模式: 模型提前停止 / 反复说"总结"

# 维度 2: Coherence
# 长文本前后是否连贯?
# 检测: 重复段落 / 前后矛盾 / 话题漂移
# 评估: sliding window 相似度 + LLM judge

# 维度 3: Factuality
# 长文本中的事实是否准确?
# 越长 → 越容易出现 hallucination
# 尤其在技术报告/综述类任务中关键

# 维度 4: Structure
# 是否有合理的组织结构?
# 检测: 段落层次 / 章节过渡 / 逻辑链条
# 4K+ 时结构崩坏是最常见失败

# 维度 5: Instruction Following
# 是否满足具体格式要求?
# 例: "写 5 章, 每章含代码示例"
# 长输出时格式遵从率显著下降

# ====== 任务类型 ======
# - 长故事 (creative writing, 5K-20K)
# - 技术报告 (structured, 3K-10K)
# - 综述文章 (survey, 5K-15K)
# - 代码文件 (code gen, 2K-8K)

# ====== 核心发现 ======
# 大部分模型在 4K+ 输出时质量急剧下降
# 典型失败模式:
#   1. Repetition cliff: 4K 后开始循环
#   2. Topic drift: 逐渐偏离原始指令
#   3. Structure collapse: 段落边界消失
#   4. Quality fade: 后半段信息密度骤降
#   5. Premature stop: 突然结束 + "总结"

标准 LLM 在 4K tokens 后输出质量急剧下降 (重复/离题/结构崩坏), LongWriter 系列模型显著延缓退化曲线

评测维度总结

Length Compliance: 模型是否能输出到目标长度, 还是提前截断? 大部分模型在 4K 后开始"强行总结"
Coherence: 滑动窗口检测重复 + LLM judge 评估连贯性. 重复是最普遍的失败模式
Structure: 长文本需要清晰的章节/段落/过渡结构. 4K+ 时结构崩坏最为显著

模型排名 (典型)

GPT-4o: 2K 内优秀, 4K+ 急速下降

Claude: 较好的长输出连贯性

GLM-4-LongWriter: 10K 仍保持可用质量

开源 7B 模型: 普遍在 2K 后崩坏

Key Insight

4K Cliff: 几乎所有未经专门长输出训练的模型, 在 4K tokens 输出后都出现质量断崖. 这是 SFT 数据长度分布的直接反映

3. LongWriter-Zero — 零样本长输出 (纯 RL 路线)

# ====== 核心思路 ======
# 不需要长输出训练数据!
# 纯 RL 驱动突破输出长度限制
# 证明: 长输出能力已存在, 只是被压制

# ====== 与 LongWriter 的区别 ======
# LongWriter:  AgentWrite 合成数据 → SFT
# LongWriter-Zero: RL reward → 直接引导
#   无需任何长输出训练样本!

# ====== Reward Design ======
def compute_reward(output, instruction):
    # 联合 reward: 长度 + 质量

    # 1. Length Reward
    # 鼓励输出更长, 但有上限防退化
    target_len = extract_target(instruction)
    len_ratio = len(output) / target_len
    if len_ratio < 1.0:
        length_r = len_ratio  # 线性奖励
    else:
        length_r = 1.0  # 达标即满分, 不鼓励过长

    # 2. Quality Reward
    # Judge model 评估连贯性 + 相关性
    quality_r = judge_model.score(
        output, instruction,
        criteria=["coherence", "relevance",
                  "no_repetition", "structure"]
    )  # score in [0, 1]

    # 联合 reward (防止只长不好)
    return length_r * 0.4 + quality_r * 0.6

# ====== RL Training ======
# Algorithm: GRPO (Group Relative Policy Opt)
# 也可用 PPO, 但 GRPO 更稳定
#
# 训练过程:
# 1. 给模型长输出任务指令
# 2. 模型尝试生成 (初始很短)
# 3. Reward 信号: 越长且质量好 → 越高分
# 4. 模型逐渐学会"长且好"的策略
#
# 关键发现:
# - 输出长度随 RL 步数 monotonically 增长
# - 从 ~2K 逐步扩展到 10K+
# - 质量不降反升 (RL 同时优化质量)
# - 证明: 模型"已有能力, 只是被 SFT 压制"
#
# ====== 意义 ======
# RL 可以零样本解锁"被压制"的长输出能力
# 无需构造任何长输出数据
# 类比: DeepSeek-R1 用 RL 解锁思维链
#       LongWriter-Zero 用 RL 解锁长输出

LongWriter-Zero 通过 RL reward (长度+质量) 逐步引导模型从 2K 扩展到 10K+, 输出长度随训练步数单调增长

LongWriter vs. LongWriter-Zero

LongWriter (SFT 路线): 需要 AgentWrite 合成 6000 条长输出数据, 然后 SFT. 数据构造成本高
LongWriter-Zero (RL 路线): 零长输出训练样本! 纯粹靠 reward 信号引导模型自主发现长输出策略
Deep Insight: 模型的长输出能力并非不存在, 而是被 SFT 阶段的短回复数据"压制". RL 可以重新释放这种潜力

类比

DeepSeek-R1: RL 解锁思维链

LongWriter-Zero: RL 解锁长输出

共同点: 都是用 RL 释放模型"已有但被 SFT 压制"的能力

Reward 设计要点

纯长度 reward → 退化为重复填充

纯质量 reward → 不会变长

联合 reward → 长且好

方法路线数据需求输出长度核心机制

Key Takeaway: LongWriter 证明"数据是瓶颈" -- 有了长输出数据就能 SFT 解锁. LongWriter-Zero 证明"RL 能零样本解锁" -- 模型本身有长输出能力, 只是被 SFT 短回复数据压制. 两者从不同角度验证了同一个假设: 输出长度限制是训练偏差, 不是模型能力瓶颈.
实践建议: 如果有计算资源做 RL → LongWriter-Zero 更优雅 (无数据依赖). 如果只能 SFT → LongWriter AgentWrite pipeline 是成熟方案. LongGenBench 用于评估任何长输出模型的质量.
